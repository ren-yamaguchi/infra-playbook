## 基本課題 A：インストールと基本操作

**A-1. EC2 への Memcached インストール**
- AL2023 の EC2 に Memcached をインストールし、`systemd` でサービス登録・自動起動を設定する
- `/etc/sysconfig/memcached` でメモリ上限・ポート・最大接続数を設定する
- `telnet localhost 11211` で接続し、`stats` コマンドでサーバー情報を取得する

**A-2. 基本コマンドの習得**
- `telnet` または `nc` 経由で以下の Memcached プロトコルコマンドを直接実行する

```
set mykey 0 60 5     # キー設定（フラグ=0、TTL=60秒、バイト数=5）
hello                # 値
get mykey            # 値の取得
delete mykey         # 削除
incr counter 1       # インクリメント
stats items          # スラブ別統計
stats slabs          # メモリ割り当て状況
flush_all            # 全キャッシュ削除（本番禁止）
```

**A-3. Memcached のアーキテクチャ理解**
- Memcached のスラブアロケーター（Slab Allocator）によるメモリ管理の仕組みを理解する
- チャンクサイズとスラブクラスの関係を `stats slabs` で確認し、メモリの断片化が発生する条件をまとめる
- LRU（Least Recently Used）によるエビクションの動作を確認する

## 基本課題 B：データ操作と TTL 管理

**B-1. TTL の設定と確認**
- TTL を 10 秒に設定したキーが時間経過後に自動削除されることを確認する
- `get` で削除されたキーへのアクセスが `END`（空）を返すことを確認する
- TTL = 0（永続）の設定と、メモリ上限到達時にエビクションが発生することを確認する

**B-2. CAS（Check And Set）操作**
- `gets` コマンドで CAS トークンを取得し、`cas` コマンドでトークンが一致する場合のみ更新する楽観的ロックを実装する
- 競合が発生した場合（別プロセスが先に更新）に `EXISTS` が返ることを確認し、リトライロジックを実装する

## 基本課題 C：分散キャッシュの動作確認

**C-1. 複数 Memcached ノードへの分散**
- 2 台の EC2 それぞれに Memcached を起動し、Python の `pymemcache` または `libmemcached` でクライアントから複数ノードを指定して接続する
- キーの分散アルゴリズム（モジュロハッシュ / 一貫性ハッシュ）の違いを理解し、ノード追加時のキャッシュ再配布の影響を比較する

```python
from pymemcache.client.hash import HashClient

client = HashClient([
    ('10.0.1.10', 11211),
    ('10.0.1.11', 11211),
])
client.set('key1', 'value1')
print(client.get('key1'))
```

**C-2. ノード障害時の動作確認**
- 一方の Memcached ノードを停止し、クライアントの動作（エラー / 別ノードへのフォールバック）を確認する
- Memcached はデータを複製しないため、ノード停止時にそのノードに格納されていたデータはすべて失われることを確認する

## 発展課題 D：Web アプリケーションとの連携

**D-1. PHP + Memcached による WordPress キャッシュ**
- PHP の `memcached` 拡張をインストールし、WordPress のオブジェクトキャッシュとして Memcached を使用する
- Redis Object Cache と Memcached Object Cache の設定の違いを比較し、キャッシュヒット率を `stats` コマンドで確認する

**D-2. セッションの Memcached 格納**
- PHP の `session.save_handler = memcached` と `session.save_path = "10.0.1.10:11211"` を設定し、セッションデータが Memcached に格納されることを確認する
- 複数 Web サーバーが共通 Memcached を参照することでセッションを共有し、ALB でラウンドロビンしてもセッションが維持されることを確認する

**D-3. Django / Flask との連携**
- Django のキャッシュバックエンドを Memcached に設定し、ビューキャッシュ（`@cache_page`）とクエリキャッシュを実装する
- `django-pylibmc` または `pymemcache` の Django バックエンドでの接続設定を確認する

## 発展課題 E：パフォーマンスチューニング

**E-1. メモリ設定の最適化**
- `-m` オプションでメモリ上限を設定し、t2.micro（1GB RAM）の場合に適切な値（例：512MB）を設定する
- `stats` の `evictions` カウンターを監視し、エビクションが多発している場合はメモリ不足のサインであることを確認する

**E-2. 接続数とスレッド数の調整**
- `-c`（最大接続数）と `-t`（スレッド数）を調整し、`ab` で負荷テストを実施してスループットの変化を計測する
- `stats` の `curr_connections` と `max_connections` を確認し、接続プールの設定と合わせる

**E-3. キャッシュヒット率の改善**
- `stats` の `get_hits` / `get_misses` からヒット率を計算するスクリプトを作成する
- TTL の設定値を調整してヒット率の変化を観察し、コンテンツ特性に合った TTL 設計を行う

## 発展課題 F：監視と運用

**F-1. stats コマンドによる監視指標の収集**
- 以下の統計情報を定期収集するスクリプトを作成し、CloudWatch カスタムメトリクスに送信する

| 指標 | 確認内容 |
|-----|---------|
| `curr_connections` | 現在の接続数 |
| `get_hits` / `get_misses` | キャッシュヒット率 |
| `evictions` | エビクション発生数（0 が理想） |
| `bytes` / `limit_maxbytes` | メモリ使用率 |
| `curr_items` | 格納アイテム数 |

**F-2. Zabbix による監視**
- Zabbix のユーザーパラメーターで Memcached の統計情報を取得し、以下のトリガーを設定する
  - メモリ使用率が 80% を超えた場合
  - エビクションが 1 分間に 100 件を超えた場合
  - 接続が確立できない場合

## 発展課題 G：セキュリティ強化

**G-1. バインドアドレスの制限**
- Memcached をデフォルトの `0.0.0.0` ではなくプライベート IP にバインドし、インターネットから直接アクセスできないようにする
- Security Group で 11211 番ポートを特定の EC2 プライベート IP のみに許可する

**G-2. SASL 認証の設定**
- Memcached の SASL 認証を有効化（`-S` オプション）し、認証なしの接続が拒否されることを確認する
- SASL 対応クライアント（`libmemcached` + `sasl2`）での接続設定を確認する

**G-3. UDP の無効化**
- Memcached は UDP でも動作するため、DDoS 増幅攻撃の踏み台になるリスクがある
- `-U 0` で UDP を無効化し、TCP のみで動作させる設定を行う（セキュリティ上の必須設定）

## 発展課題 H：Redis との比較と使い分け

**H-1. 比較表の作成**

| 観点 | Memcached | Redis |
|------|----------|-------|
| データ型 | 文字列のみ | String / List / Hash / Set / Sorted Set 等 |
| 永続化 | なし | RDB / AOF で対応 |
| レプリケーション | なし（クライアント側で分散） | Primary / Replica |
| クラスタリング | クライアント側ハッシュ | Redis Cluster（サーバー側） |
| マルチスレッド | ○（複数スレッド） | シングルスレッド（6.0 以降は I/O マルチスレッド） |
| メモリ効率 | 高（シンプルな構造） | 中（多機能なためオーバーヘッドあり） |
| Pub/Sub | なし | あり |
| スクリプト | なし | Lua スクリプト対応 |
| 用途 | シンプルな KV キャッシュ | 多目的（キャッシュ・セッション・キュー・Pub/Sub） |

**H-2. 用途別の選択基準**
- 単純な文字列キャッシュで最大のスループットが必要 → Memcached
- セッション管理・ランキング・Pub/Sub・データ構造が必要 → Redis
- データの永続化・レプリケーションが必要 → Redis
- マルチスレッドで CPU コアを最大活用したい → Memcached
