## 2. 目的・概要

### 2-1. 目的

> 本手順書では，Tomcat×2台（AP1／AP2）でアップロードディレクトリを共有するため，DBサーバーに同居するNFSv4サーバーを構築し，AP1／AP2 からNFSマウントできる状態にする手順について説明する．
>
> 本手順書のゴールは「**AP1／AP2 の両方から同じ共有ディレクトリにアクセスでき，どちらに書き込んだファイルも他方から読み出せる状態**」とする．
>
> ファイル共有先のディレクトリ構造・所有者・パーミッションのアプリケーション要件は，個別アプリ手順書（例：`war-deploy-migration.md`）で実施する．

### 2-2. 構成概要（アーキテクチャ）

```
┌────────────────────────── VPC ──────────────────────────┐
│                                                          │
│  [Private-AZ2: DB／NFS同居サーバー]                       │
│    ├─ Amazon Linux 2023                                  │
│    ├─ PostgreSQL 15（`postgresql-server.md` で構築済）      │
│    ├─ nfs-utils（NFSv4サーバー）                           │
│    │     └─ TCP/2049                                     │
│    │     └─ /etc/exports でエクスポート設定              │
│    │                                                     │
│    └─ /exports/<アプリケーションデータパス>                 │
│         （ルートボリューム上のディレクトリ）                  │
│                                                          │
│                ▲                  ▲                      │
│                │ TCP/2049         │ TCP/2049              │
│                │                  │                      │
│  [Private-AZ2: AP1]      [Private-AZ4: AP2]              │
│    └─ Tomcat              └─ Tomcat                       │
│       └─ /mnt/<アプリ      └─ /mnt/<アプリ                │
│          ケーションデータ      ケーションデータ              │
│          パス>                 パス>                       │
│         （NFSマウント）       （NFSマウント）              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 2-3. 完成イメージ（ゴール定義）

- [ ] DB／NFS同居サーバーに `nfs-utils` がインストールされている
- [ ] `nfs-server.service` が `active (running)` かつ自動起動有効
- [ ] 2049番ポートでLISTENしている
- [ ] `/exports/<アプリケーションデータパス>` が存在する
- [ ] `/etc/exports` に共有設定が登録されている
- [ ] `exportfs -v` でエクスポート状態を確認できる
- [ ] AP1 から `/mnt/<アプリケーションデータパス>` がマウントされている
- [ ] AP2 から `/mnt/<アプリケーションデータパス>` がマウントされている
- [ ] AP1 で作成したファイルがAP2から見える（共有確認）
- [ ] AP1／AP2 ともOS再起動後も自動マウントされる（`/etc/fstab` 登録済み）

## 4. 構築手順（詳細）

> **注意事項**
> - コマンド中の `<山カッコ>` は自分の環境の値（パラメータ定義表の値）に置き換えること
> - 各Stepの見出し末尾に **【実施対象：●●】** を明示しているので，対象のサーバーで実施すること
> - 作業対象は **NFSサーバー（DBサーバー同居）** と **クライアント（AP1／AP2）** の2系統に分かれる

------------------------------

### Step 1：nfs-utils のインストール【実施対象：DB／NFS同居サーバー】

**目的：** NFSv4サーバー機能を提供する `nfs-utils` パッケージをインストールする

#### 操作手順

```bash
# rootユーザーにスイッチ
sudo su -

# パッケージを最新化
dnf update -y

# nfs-utilsをインストール
dnf install -y nfs-utils

# インストール確認
rpm -q nfs-utils
```

> **期待する結果：** `nfs-utils-x.x.x-x.amzn2023.x86_64` のようなパッケージ名が表示される．

------------------------------

### Step 2：共有ディレクトリの作成【実施対象：DB／NFS同居サーバー】

**目的：** NFSでエクスポートする共有ディレクトリを作成する

#### 操作手順

```bash
# 共有ディレクトリ作成
mkdir -p /exports/<アプリケーションデータパス>

# 所有者の設定（Tomcat実行UID／GIDが確定している場合）
chown -R <アプリ実行UID>:<アプリ実行GID> /exports/<アプリケーションデータパス>

# パーミッションを設定（オーナー書込可，グループ読取可）
chmod 755 /exports/<アプリケーションデータパス>

# 確認
ls -ld /exports/<アプリケーションデータパス>
```

> **期待する結果：** `drwxr-xr-x ... <UID> <GID> ... /exports/<アプリケーションデータパス>` が表示される．

> **重要：** Tomcat実行ユーザーのUID／GIDは `war-deploy-migration.md` で確定する．Tomcat構築前に本手順書を実施する場合は，一旦 `root:root` でディレクトリだけ作成し，後でUID／GIDを揃える．

> **補足：** NFSv4 では `root_squash`（後続Step）によりクライアントの `root` は `nobody` にマップされる．Tomcatアプリは通常 `tomcat` ユーザーで動くため，AP1／AP2 の `tomcat` UID／GID と NFSサーバー側のUID／GID を一致させる必要がある．

------------------------------

### Step 3：/etc/exports の設定【実施対象：DB／NFS同居サーバー】

**目的：** 共有ディレクトリをNFSでエクスポートする設定を `/etc/exports` に追加する

#### 操作手順

```bash
# /etc/exports を編集
vi /etc/exports
```

以下の行を追加：

```
/exports/<アプリケーションデータパス>  <許可CIDR>(rw,sync,root_squash,no_subtree_check,fsid=0)
```

> **重要：** `<許可CIDR>` と `(...)` の間に**スペースを入れない**こと．スペースを入れると挙動が変わる（`root_squash` を含む全クライアント許可になる）．

#### オプションの意味

| オプション | 説明 |
|---|---|
| `rw` | 読み書き許可 |
| `sync` | 同期書き込み（データ整合性優先） |
| `root_squash` | クライアントの `root` を `nobody`（UID 65534）にマップ．**推奨**（セキュリティ向上） |
| `no_subtree_check` | サブツリーチェック無効．パフォーマンス向上 |
| `fsid=0` | NFSv4 のルートエクスポート指定 |

#### 確認

```bash
cat /etc/exports
```

> **期待する結果：** 追加した行が表示される．

------------------------------

### Step 4：NFSサービスの起動・自動起動設定【実施対象：DB／NFS同居サーバー】

**目的：** NFSサーバーを起動し，OS再起動時にも自動起動する設定にする

#### 操作手順

```bash
# NFSサービスの起動・自動起動有効化
systemctl enable --now nfs-server.service

# 状態確認
systemctl status nfs-server.service --no-pager
systemctl is-enabled nfs-server.service
```

> **期待する結果：** `Active: active (running)` および `enabled`．

#### エクスポート反映の確認

```bash
exportfs -v
```

> **期待する結果：** 以下のように設定がリスト表示される：
>
> ```
> /exports/<アプリケーションデータパス>
>     <許可CIDR>(rw,wdelay,root_squash,no_subtree_check,sec=sys,rw,secure,root_squash,no_all_squash,fsid=0)
> ```

#### LISTEN確認

```bash
ss -tlnp | grep :2049
```

> **期待する結果：** `0.0.0.0:2049` がLISTEN．

> **注意：** 起動失敗時は `journalctl -u nfs-server -n 50` でログを確認．`/etc/exports` の構文ミス・ディレクトリ未作成が主な原因．

------------------------------

### Step 5：NFSクライアント側パッケージインストール【実施対象：AP1／AP2】

**目的：** クライアント側でもNFSマウントに必要な `nfs-utils` をインストールする

#### 操作手順（AP1／AP2 の両方で実施）

```bash
sudo su -

dnf update -y
dnf install -y nfs-utils

rpm -q nfs-utils
```

> **期待する結果：** パッケージ名が表示される．

------------------------------

### Step 6：マウントポイントの作成【実施対象：AP1／AP2】

**目的：** クライアント側でNFSマウント先のディレクトリを作成する

#### 操作手順（AP1／AP2 の両方で実施）

```bash
# マウントポイント作成
mkdir -p /mnt/<アプリケーションデータパス>

# 確認
ls -ld /mnt/<アプリケーションデータパス>
```

> **期待する結果：** `drwxr-xr-x ... root root ... /mnt/<アプリケーションデータパス>` が表示される（NFSマウント後にサーバー側の所有者／パーミッションが反映される）．

------------------------------

### Step 7：手動マウントによる接続確認【実施対象：AP1／AP2】

**目的：** `/etc/fstab` で永続化する前に，手動マウントで疎通を確認する

#### 操作手順（AP1／AP2 の両方で実施）

```bash
# 手動マウント（NFSv4）
mount -t nfs4 <NFSサーバーのプライベートIP>:/<アプリケーションデータパス> /mnt/<アプリケーションデータパス>

# マウント状態確認
mount | grep <アプリケーションデータパス>
df -hT | grep <アプリケーションデータパス>
```

> **期待する結果：**
>
> ```
> <NFSサーバーのプライベートIP>:/<アプリケーションデータパス> on /mnt/<アプリケーションデータパス> type nfs4 (rw,relatime,...)
> ```

> **重要：** NFSv4 で `fsid=0` を使うエクスポートは，クライアント側でパスから `/exports` プレフィックスを省略する必要がある．具体的には `<サーバーIP>:/<アプリケーションデータパス>` と指定する（`/exports/...` ではない）．

#### マウント失敗時の確認

```bash
# RPC統計確認
rpcinfo -p <NFSサーバーのプライベートIP>

# ファイアウォール（NFSサーバー側のSG）で 2049 が開いているか確認
# AWSコンソールで対象SGのインバウンドルールを目視
```

------------------------------

### Step 8：`/etc/fstab` への登録（永続マウント）【実施対象：AP1／AP2】

**目的：** OS再起動時にも自動的にNFSマウントされるように `/etc/fstab` に登録する

#### 操作手順（AP1／AP2 の両方で実施）

```bash
# /etc/fstab を編集
vi /etc/fstab
```

以下の行を追加：

```
<NFSサーバーのプライベートIP>:/<アプリケーションデータパス>  /mnt/<アプリケーションデータパス>  nfs4  defaults,_netdev,nofail  0  0
```

#### オプションの意味

| オプション | 説明 |
|---|---|
| `defaults` | デフォルトオプション |
| `_netdev` | ネットワーク利用デバイスを示す．ネットワーク確立後にマウントを試行 |
| `nofail` | マウント失敗時もOS起動を継続．**重要**（NFSサーバーダウン時のブート失敗を回避） |
| dump `0` | バックアップ要否．NFSは `0` |
| fsck pass `0` | OS起動時のfsck順序．NFSは `0` |

#### 反映確認

```bash
# 一旦アンマウント
umount /mnt/<アプリケーションデータパス>

# fstabから再マウント
mount -a

# マウント状態確認
df -hT | grep <アプリケーションデータパス>
```

> **期待する結果：** `mount -a` がエラーなく完了し，マウントが復元される．

> **重要：** `/etc/fstab` の記述ミスがあると次回OS起動時にブートが失敗する可能性がある．`nofail` を必ず付けること．加えて，`mount -a` でエラーがないことを確認してから再起動すること．

## 6. トラブルシューティング

------------------------------

#### エラー①：`nfs-server.service: Failed to start`

**原因：**

- `/etc/exports` の構文ミス
- 共有ディレクトリ（`/exports/<アプリケーションデータパス>`）が未作成

**対処法：**

```bash
# ログ確認
journalctl -u nfs-server.service -n 50 --no-pager

# /etc/exports の構文確認
exportfs -ra
# → エラーメッセージが表示される場合は該当行を修正

# 共有ディレクトリの存在確認
ls -ld /exports/<アプリケーションデータパス>
```

------------------------------

#### エラー②：クライアントから `mount.nfs4: Connection timed out`

**原因：**

- NFSサーバーのSG で 2049/TCP が許可されていない
- NFSサーバーが起動していない
- ネットワーク経路の問題

**対処法：**

```bash
# クライアントから疎通確認
ping -c 3 <NFSサーバーのプライベートIP>

# NFSサーバー側で 2049 LISTEN確認
ss -tlnp | grep :2049
```

> AWSコンソールで対象SGのインバウンドルールを目視確認．`TCP 2049` が許可されているか，ソースに `<許可CIDR>` または AP1／AP2 のSGが含まれているか確認．

------------------------------

#### エラー③：クライアントから `mount.nfs4: access denied by server`

**原因：**

- `/etc/exports` の許可CIDRがクライアントのIPを含んでいない
- `/etc/exports` 編集後に `exportfs -ra` を実行していない

**対処法：**

NFSサーバー側で：

```bash
# /etc/exports 確認
cat /etc/exports

# クライアントIPが <許可CIDR> に含まれるか確認
# 例：<許可CIDR> が 10.1.0.0/16 で，AP1 が 10.1.16.50 なら含まれる

# /etc/exports 編集後は反映が必要
exportfs -ra

# 反映確認
exportfs -v
```

------------------------------

#### エラー④：マウントできるが書き込みで `Permission denied`

**原因：**

- `root_squash` により `root` での書き込みが拒否されている
- 共有ディレクトリの所有者／パーミッションがTomcat実行UID／GIDと不一致

**対処法：**

NFSサーバー側で：

```bash
# 共有ディレクトリの所有者を確認
ls -ld /exports/<アプリケーションデータパス>

# Tomcat実行ユーザーのUID／GIDに合わせる
chown -R <アプリ実行UID>:<アプリ実行GID> /exports/<アプリケーションデータパス>
chmod 755 /exports/<アプリケーションデータパス>
```

クライアント側で：

```bash
# クライアント側のTomcat実行ユーザーのUID／GIDを確認
id tomcat

# UID／GIDが一致しているか確認．不一致の場合は `usermod` / `groupmod` で揃える
```

> **重要：** NFSはサーバー側のUID／GIDをクライアント側でも同じものとして扱う．**両者でUID／GIDが一致している必要がある**．

------------------------------

#### エラー⑤：OS再起動後に `/mnt/<アプリケーションデータパス>` がマウントされていない

**原因：**

- `/etc/fstab` の記述ミス
- NFSサーバーがクライアント起動時にまだ未起動

**対処法：**

```bash
# /etc/fstab の記述確認
grep <アプリケーションデータパス> /etc/fstab

# 手動マウントを試す
mount -a

# エラー内容確認
dmesg | tail -20
```

> **補足：** `nofail` オプションがあっても，ネットワーク確立タイミングの問題でマウントに失敗することがある．その場合は `_netdev` オプションが付いているか，systemd の `nfs-client.target` が enabled か確認する．

------------------------------

|-----------|------|
| NFSサーバーログ | `journalctl -u nfs-server` |
| `rpc.mountd` ログ | `journalctl -u nfs-mountd`（NFSv4 では通常不要） |

### B. 設定ファイル解説

**`/etc/exports`（NFSサーバー）**

書式：

```
<共有ディレクトリ>  <許可CIDRまたはホスト>(オプション)
```

主なオプション：

| オプション | 説明 |
|---|---|
| `rw` | 読み書き許可 |
| `ro` | 読み取り専用 |
| `sync` | 同期書き込み．データ整合性優先（推奨） |
| `async` | 非同期書き込み．性能優先（データ損失リスクあり） |
| `root_squash` | クライアントの `root` を `nobody`（UID 65534）にマップ．**セキュリティ向上のため推奨** |
| `no_root_squash` | `root` のままアクセス．特殊用途のみ |
| `all_squash` | 全UIDを `nobody` にマップ．完全匿名化 |
| `no_subtree_check` | サブツリーチェック無効．パフォーマンス向上 |
| `fsid=0` | NFSv4 のルートエクスポート指定 |

> **重要：** `<許可CIDR>(オプション)` の間にスペースを入れない．スペースを入れると `<許可CIDR>` と `(オプション)` が別エントリと解釈され，挙動が変わる．

**`/etc/fstab`（NFSクライアント）**

書式：

```
<デバイス>  <マウントポイント>  <FS種別>  <オプション>  <dump>  <fsck>
```

NFSv4 マウント例：

```
<NFSサーバーのIP>:/<パス>  /mnt/<パス>  nfs4  defaults,_netdev,nofail  0  0
```

| 項目 | 値 |
|---|---|
| FS種別 | `nfs4`（NFSv4 を明示） |
| オプション | `defaults,_netdev,nofail` 推奨 |
| dump | `0`（NFSはバックアップ対象外） |
| fsck | `0`（NFSはfsck対象外） |

### C. 用語解説

| 用語 | 説明 |
|------|------|
| NFS | Network File System．ネットワーク経由でファイルシステムを共有するプロトコル． |
| NFSv4 | NFSのバージョン4．`TCP/2049` のみで完結．`portmap` 不要．セキュア． |
| NFSv3 | NFSのバージョン3．`portmap` / `mountd` などの追加ポートが必要．レガシー用途． |
| エクスポート | NFSサーバー側で共有を公開する設定．`/etc/exports` で定義． |
| `exportfs` | エクスポート設定を管理するコマンド． |
| `fsid=0` | NFSv4 のルートエクスポート指定．これがあるとクライアントはサーバー側パスから `/exports` プレフィックスを省略する． |
| `root_squash` | クライアントの `root` 操作を `nobody` にマップする．セキュリティ向上のための仕組み． |
| UID／GID | UNIX系OSのユーザーID／グループID．NFSはサーバー・クライアント間でUID／GID値を共有する． |
| `nofail` | `/etc/fstab` オプション．マウント失敗時もOS起動を継続させる． |
| `_netdev` | `/etc/fstab` オプション．ネットワークデバイスであることを示す．ネットワーク確立後にマウントを試行． |

### D. 補足解説

#### D-1. なぜNFSv4を採用するか？

| 観点 | NFSv4 | NFSv3 |
|---|---|---|
| 使用ポート | TCP/2049 のみ | TCP/2049 + portmap/mountd（動的） |
| ファイアウォール設定 | シンプル | 複雑（動的ポートを開ける必要あり） |
| セキュリティ | Kerberos対応・統合認証可能 | 基本的なIPベース認証のみ |
| パフォーマンス | 改善されたキャッシュ・ロック機構 | レガシー |
| 互換性 | Linux/Unix系で広く対応 | 同じく広く対応 |

本手順書ではセキュリティとファイアウォール設定のシンプルさから **NFSv4を採用**．

#### D-2. なぜ `root_squash` を有効にするか？

`root_squash` は，クライアントの `root` ユーザーがNFS経由でアクセスする際，サーバー側では `nobody`（UID 65534）として扱う仕組み．

| 観点 | `root_squash`（推奨） | `no_root_squash` |
|---|---|---|
| セキュリティ | **高**．クライアント侵害時もサーバー側 `root` 権限は得られない | 低．クライアント `root` がそのままサーバー側でも `root` |
| 運用の便利さ | 共有ディレクトリは Tomcat 実行ユーザー（非root）でアクセスする前提 | `root` で何でもできてしまう |
| 一般的な推奨 | **本番環境で推奨** | 開発・検証の限定用途のみ |

Tomcatアプリは通常 `tomcat` ユーザーで動作するため，`root_squash` 有効でも実用上の問題は発生しない．

#### D-3. UID／GID の一致が必要な理由

NFSはサーバー・クライアント間でUID／GID値を**そのまま受け渡す**．名前ベース（`tomcat`）ではなく，数値ベース（UID 1001 等）で識別される．

例：
- NFSサーバー側で共有ディレクトリの所有者が `tomcat`（UID 1001）
- AP1 の `tomcat` ユーザーが UID 1001 → OK，書き込み可能
- AP2 の `tomcat` ユーザーが UID 1002 → NG，書き込み拒否される

これを回避するため，AP1／AP2 の `tomcat` ユーザーを**同じUID／GID**で作成する必要がある．`war-deploy-migration.md` でUID／GIDを明示的に揃える手順を実施する．

#### D-4. ストレージ方針について

本演習では，DBサーバーとNFSサーバーを **同一EC2** に同居させ，**ルートボリューム50GiB** に PostgreSQL データと NFS 共有データの両方を配置する．

| 観点 | 同居構成（本演習） | 分離構成（本番想定） |
|---|---|---|
| EC2台数 | 1台 | 2台以上 |
| ストレージ | ルートボリューム1枚 | OS用・DB用・NFS用に分離 |
| コスト | 安い | 高い |
| 障害分離 | 低い（同居EC2障害で両方停止） | 高い |
| 推奨用途 | 検証・演習 | 本番 |

本演習では学習目的のため同居構成を採用しているが，本番運用では役割ごとにEC2を分離し，データ用にEBSを別ボリュームとして追加することが望ましい．

#### D-5. NFSサーバーが停止した場合のクライアント挙動

NFSv4 でマウント中にサーバーが停止すると，クライアント側のプロセスがハングする場合がある．対処法：

| マウントオプション | 挙動 |
|---|---|
| `hard`（デフォルト） | サーバー復旧まで永久待機．プロセスがハング |
| `soft` | タイムアウト後にI/Oエラーで失敗．データ損失リスクあり |
| `intr` | `hard` だが `Ctrl+C` 等で中断可能 |

本手順書では `defaults`（`hard`）を採用．本番運用ではアプリケーション特性に応じて検討．
