## 2. 目的・概要

### 2.1 目的

本手順書は，本手順書ライブラリの**土台となるAWS基盤**を構築するための手順をまとめたものです．

以下の14手順書を実行する**前提条件**として，本手順書を最初に実施します．

* `nsd-private-redundancy.md`
* `chrony-internal-ntp.md`
* `postgresql-server.md`
* `tomcat-basic.md`
* `nginx-reverse-proxy.md`
* `postfix-smtp-relay.md`
* `nsd-public-letsencrypt.md`
* `alb-acm-https.md`
* `zabbix-db-postgresql.md`
* `zabbix-server.md`
* `zabbix-agent2.md`
* `postgresql-migration.md`
* `war-deploy-migration.md`
* `modsecurity-migration.md`

本手順書では，以下を実施します．

* VPC・サブネット（Public×2 / Private×2）の作成
* インターネットゲートウェイ（IGW）の作成・アタッチ
* NATゲートウェイの作成
* ルートテーブルの設定
* 踏み台サーバーの構築
* EC2インスタンスの作成（共通手順）
* EBS追加ボリュームのアタッチ・マウント（共通手順）

## 3. 前提条件

### 3.1 前提手順書

本手順書は，本ライブラリの最初の手順書です．前提手順書はありません．

### 3.2 環境要件

| 項目         | 要件                                  |
| ---------- | ----------------------------------- |
| AWSアカウント   | 管理者権限相当（VPC・EC2の作成・変更権限）          |
| 作業端末       | ブラウザ（AWSマネジメントコンソールアクセス用）          |
| 作業端末       | SSHクライアント（踏み台サーバーへのログイン用）        |
| 作業端末       | グローバルIPが取得できること（SG設定用）              |

### 3.3 想定される影響

* 既存のVPCやサブネットのCIDRと重複すると，後でVPCピアリングなどができなくなる可能性があります．
* NATゲートウェイ・EIPの利用は時間課金が発生します．
* 踏み台サーバーのSGをマイIP制限せず公開すると，不正アクセスのリスクがあります．

### 3.4 ロールバック可能性

ロールバックは可能です．削除順序は以下です．

```text
EC2 → NATGW → EIP → サブネット → ルートテーブル → IGW → VPC
```

### 3.5 事前準備チェックリスト

* [ ] AWSアカウントに管理者権限相当でログインできる
* [ ] 作業端末のグローバルIPを把握している
* [ ] VPC CIDRが既存ネットワークと重複しないことを確認した
* [ ] パラメータ定義表（2.4）に値を記入した
* [ ] キーペア（`.pem`）の保管場所を確保している

## 4. 構築手順

### 4.1 VPCの作成【実施対象：AWSコンソール】

実施内容：VPC（仮想プライベートネットワーク）を作成します．

操作手順：

1. AWSマネジメントコンソールにログインし，右上のリージョンが `<リージョン>` になっていることを確認します．
2. 上部の検索欄に `VPC` と入力し，VPCサービスを開きます．
3. 左メニューから「VPC」を選択し，右上の「**VPCを作成**」をクリックします．
4. 以下を入力します（その他はデフォルト）．
   - 作成するリソース：`VPCのみ`
   - 名前タグ：`<VPC名>`
   - IPv4 CIDR：`<VPC CIDR>`
   - テナンシー：`デフォルト`
5. 右下の「**VPCを作成**」をクリックします．

期待される結果：

VPC一覧に作成したVPCが表示され，状態が「使用可能」になります．

注意点・コメント：

VPC作成後に「DNSホスト名」「DNS解決」を有効化します．

1. 作成したVPCを選択 →「アクション ▼」→「**VPC設定を編集**」
2. 「DNSホスト名を有効化」「DNS解決を有効化」にチェック → 保存

------------------------------

### 4.2 サブネットの作成【実施対象：AWSコンソール】

実施内容：Public×2，Private×2 の合計4つのサブネットを作成します．

操作手順（4回繰り返す）：

1. VPC画面の左メニューから「**サブネット**」を選択します．
2. 右上の「**サブネットを作成**」をクリックします．
3. VPC ID：`<VPC名>` を選択します．
4. 「サブネット1」に以下を入力します．

**サブネット1：Public-AZ1**

| 項目                 | 値                   |
| ------------------ | ------------------- |
| サブネット名              | Public-AZ1          |
| アベイラビリティーゾーン       | `<AZ1>`             |
| IPv4サブネットCIDRブロック | `<Public-AZ1 CIDR>` |

**サブネット2：Private-AZ2**

| 項目                 | 値                    |
| ------------------ | -------------------- |
| サブネット名              | Private-AZ2          |
| アベイラビリティーゾーン       | `<AZ2>`              |
| IPv4サブネットCIDRブロック | `<Private-AZ2 CIDR>` |

**サブネット3：Public-AZ3**

| 項目                 | 値                   |
| ------------------ | ------------------- |
| サブネット名              | Public-AZ3          |
| アベイラビリティーゾーン       | `<AZ3>`             |
| IPv4サブネットCIDRブロック | `<Public-AZ3 CIDR>` |

**サブネット4：Private-AZ4**

| 項目                 | 値                    |
| ------------------ | -------------------- |
| サブネット名              | Private-AZ4          |
| アベイラビリティーゾーン       | `<AZ4>`              |
| IPv4サブネットCIDRブロック | `<Private-AZ4 CIDR>` |

5. 右下の「**サブネットを作成**」をクリックします．

期待される結果：

サブネット一覧に4つのサブネットが「使用可能」状態で表示されます．

------------------------------

### 4.3 Publicサブネットのパブリックaddress自動割当を有効化【実施対象：AWSコンソール】

実施内容：Public-AZ1とPublic-AZ3に配置するEC2に自動でパブリックIPv4を付与する設定を行います．

操作手順（2回繰り返す）：

1. サブネット一覧で `Public-AZ1` を選択します．
2. 「アクション ▼」→「**サブネット設定を編集**」をクリックします．
3. 「パブリックIPv4アドレスの自動割り当てを有効化」にチェックを入れます．
4. 「保存」をクリックします．
5. 同じ操作を `Public-AZ3` でも実施します．

期待される結果：

両サブネットの「自動割り当てパブリックIPv4アドレス」が「はい」になります．

注意点・コメント：

Privateサブネット（AZ2，AZ4）はこの設定を **無効のまま** にします．

------------------------------

### 4.4 インターネットゲートウェイの作成とアタッチ【実施対象：AWSコンソール】

実施内容：VPCをインターネットに接続するためのIGWを作成し，VPCにアタッチします．

操作手順：

1. VPC画面の左メニューから「**インターネットゲートウェイ**」を選択します．
2. 右上の「**インターネットゲートウェイの作成**」をクリックします．
3. 名前タグ：`<VPC名>-igw` を入力し，「**インターネットゲートウェイの作成**」をクリックします．
4. 作成直後の画面で「**VPCへアタッチ**」をクリックします．
5. 使用可能なVPCで `<VPC名>` を選択し，「**インターネットゲートウェイのアタッチ**」をクリックします．

期待される結果：

IGWの状態が「Attached」になります．

------------------------------

### 4.5 NATゲートウェイの作成【実施対象：AWSコンソール】

実施内容：Privateサブネット内のEC2が外部（yum/dnfリポジトリなど）にアクセスできるよう，NATゲートウェイを作成します．

操作手順：

1. VPC画面の左メニューから「**NATゲートウェイ**」を選択します．
2. 右上の「**NATゲートウェイを作成**」をクリックします．
3. 以下を入力します．
   - 名前：`<VPC名>-natgw`
   - サブネット：`Public-AZ1`
   - 接続タイプ：`パブリック`
   - Elastic IP割り当てID：「**Elastic IPを割り当て**」ボタンをクリックして新規取得
4. 右下の「**NATゲートウェイを作成**」をクリックします．

期待される結果：

NATゲートウェイの状態が「Available」になります（数分かかります）．

注意点・コメント：

NATゲートウェイは時間課金されます．コスト最適化のため，本番では各AZに1つずつ配置するのが理想ですが，本手順では検証用に1つのみ作成します．

------------------------------

### 4.6 ルートテーブルの作成と関連付け【実施対象：AWSコンソール】

実施内容：Public用とPrivate用の2つのルートテーブルを作成し，経路を設定してサブネットに関連付けます．

#### 4.6.1 Public用ルートテーブル

操作手順：

1. VPC画面の左メニューから「**ルートテーブル**」を選択します．
2. 右上の「**ルートテーブルを作成**」をクリックします．
3. 以下を入力します．
   - 名前：`<VPC名>-rt-public`
   - VPC：`<VPC名>`
4. 「**ルートテーブルを作成**」をクリックします．
5. 作成したルートテーブルを選択 →「ルート」タブ →「**ルートを編集**」
6. 「ルートを追加」をクリックし，以下を入力します．
   - 送信先：`0.0.0.0/0`
   - ターゲット：`Internet Gateway` → `<VPC名>-igw`
7. 「変更を保存」をクリックします．
8. 「**サブネットの関連付け**」タブ →「**サブネットの関連付けを編集**」
9. `Public-AZ1` と `Public-AZ3` にチェック → 「関連付けを保存」

#### 4.6.2 Private用ルートテーブル

操作手順：

1. 同じ画面で「**ルートテーブルを作成**」をクリックします．
2. 以下を入力します．
   - 名前：`<VPC名>-rt-private`
   - VPC：`<VPC名>`
3. 「**ルートテーブルを作成**」をクリックします．
4. 作成したルートテーブルを選択 →「ルート」タブ →「**ルートを編集**」
5. 「ルートを追加」をクリックし，以下を入力します．
   - 送信先：`0.0.0.0/0`
   - ターゲット：`NAT Gateway` → `<VPC名>-natgw`
6. 「変更を保存」をクリックします．
7. 「**サブネットの関連付け**」タブ →「**サブネットの関連付けを編集**」
8. `Private-AZ2` と `Private-AZ4` にチェック → 「関連付けを保存」

期待される結果：

* `<VPC名>-rt-public` のルートに `0.0.0.0/0 → <VPC名>-igw` が登録され，Public-AZ1とPublic-AZ3に関連付けられている．
* `<VPC名>-rt-private` のルートに `0.0.0.0/0 → <VPC名>-natgw` が登録され，Private-AZ2とPrivate-AZ4に関連付けられている．

### 4.7 キーペアの作成【実施対象：AWSコンソール】

実施内容：踏み台サーバーおよび後続EC2へのSSH接続用キーペアを作成します．

操作手順：

1. EC2画面の左メニューから「**キーペア**」を選択します．
2. 右上の「**キーペアを作成**」をクリックします．
3. 以下を入力します．
   - 名前：`<キーペア名>`
   - キーペアのタイプ：`RSA`
   - プライベートキーファイル形式：`.pem`
4. 「**キーペアを作成**」をクリックします．
5. 自動ダウンロードされた `<キーペア名>.pem` を安全な場所に保管します．

注意点・コメント：

`.pem` ファイルは再ダウンロード不可です．紛失するとEC2にログインできなくなります．

作業端末（Linux/Mac）では，パーミッションを `600` に変更します．

```bash
chmod 600 ~/Downloads/<キーペア名>.pem
```

### 4.8 踏み台サーバー用セキュリティグループの作成【実施対象：AWSコンソール】

実施内容：踏み台サーバー用のSGを作成し，作業端末からのSSHのみを許可します．

操作手順：

1. VPC画面の左メニューから「**セキュリティグループ**」を選択します．
2. 右上の「**セキュリティグループを作成**」をクリックします．
3. 以下を入力します．
   - セキュリティグループ名：`<VPC名>-sg-bastion`
   - 説明：`Bastion server SG`
   - VPC：`<VPC名>`
4. インバウンドルールに以下を追加します．

| タイプ | プロトコル | ポート範囲 | ソース            | 説明           |
| --- | ----- | ----- | -------------- | ------------ |
| SSH | TCP   | 22    | `<作業端末のIP>/32` | 作業端末からのSSH接続 |

5. アウトバウンドルールはデフォルト（すべて許可）のままにします．
6. 「**セキュリティグループを作成**」をクリックします．

期待される結果：

セキュリティグループ一覧に `<VPC名>-sg-bastion` が作成されます．

### 4.9 踏み台サーバー（EC2）の作成【実施対象：AWSコンソール】

実施内容：Public-AZ1に踏み台サーバーを構築します．

操作手順：

1. EC2画面の左メニューから「**インスタンス**」を選択します．
2. 「**インスタンスを起動**」をクリックします．
3. 以下を入力します．

| 項目              | 値                            |
| --------------- | ---------------------------- |
| 名前              | `<踏み台サーバー名>`                 |
| AMI             | Amazon Linux 2023            |
| アーキテクチャ         | 64ビット（x86）                   |
| インスタンスタイプ       | `t3.micro`                   |
| キーペア            | `<キーペア名>`                    |
| VPC             | `<VPC名>`                     |
| サブネット            | `Public-AZ1`                 |
| パブリックIPの自動割り当て   | 有効化                          |
| ファイアウォール（SG）    | 「既存のSGを選択」→ `<VPC名>-sg-bastion` |
| ストレージ            | 8GiB（gp3）                    |

4. 「**インスタンスを起動**」をクリックします．

期待される結果：

* インスタンスの状態が「実行中」になる．
* ステータスチェックが「2/2のチェックに合格」になる．
* パブリックIPv4アドレスが割り当てられている．

### 4.10 踏み台サーバーへのSSH接続確認【実施対象：作業端末】

実施内容：踏み台サーバーへのSSH接続を確認します．

操作手順：

```bash
ssh -i ~/Downloads/<キーペア名>.pem ec2-user@<踏み台サーバーのパブリックIP>
```

期待される結果：

```text
   ,     #_
   ~\_  ####_        Amazon Linux 2023
  ~~  \_#####\

```

注意点・コメント：

初回接続時に「The authenticity of host ... can't be established. ...」と表示されたら，`yes` と入力して進めます．

### 4.11 EC2インスタンス作成の共通手順【実施対象：AWSコンソール】（後続手順書から参照される）

実施内容：後続の各MW手順書で必要となる **EC2インスタンス作成の共通手順** を以下にまとめます．

操作手順（後続手順書ごとに繰り返す）：

1. EC2画面 →「**インスタンスを起動**」をクリックします．
2. 以下を後続手順書のパラメータに従って入力します．

| 項目                | 入力内容                                              |
| ----------------- | ------------------------------------------------- |
| 名前                | 後続手順書で指定された名前                                     |
| AMI               | Amazon Linux 2023                                 |
| アーキテクチャ           | 64ビット（x86）                                        |
| インスタンスタイプ         | 後続手順書で指定（例：t3.small / t3.medium）                 |
| キーペア              | `<キーペア名>`                                         |
| VPC               | `<VPC名>`                                          |
| サブネット             | 後続手順書で指定（Public-AZ1 / Private-AZ2 / Public-AZ3 / Private-AZ4） |
| パブリックIPの自動割り当て    | サブネット設定に従う                                        |
| ファイアウォール（SG）      | 後続手順書で指定（新規作成 or 既存SGを選択）                         |

3. 「**インスタンスを起動**」をクリックします．

注意点・コメント：

Privateサブネットに配置するEC2は，パブリックIPが付与されないため，踏み台サーバー経由でSSH接続します．

```bash
# 踏み台サーバーにSSH接続後
ssh -i ~/<キーペア名>.pem ec2-user@<対象EC2のプライベートIP>
```

（事前に踏み台サーバーへ `<キーペア名>.pem` を `scp` でアップロードしておく必要があります．）

### 4.12 EBS追加ボリュームのアタッチ・マウント共通手順【実施対象：AWSコンソール → 対象EC2】（後続手順書から参照される）

実施内容：PostgreSQLサーバーなどでデータ用に追加EBSボリュームを使用する場合の，アタッチからマウントまでの共通手順を以下にまとめます．

#### 4.12.1 EBSボリュームの作成とアタッチ【実施対象：AWSコンソール】

操作手順：

1. EC2画面の左メニューから「**ボリューム**」を選択します．
2. 右上の「**ボリュームを作成**」をクリックします．
3. 以下を入力します．
   - ボリュームタイプ：`gp3`
   - サイズ：後続手順書で指定（例：50 GiB）
   - アベイラビリティーゾーン：**対象EC2と同じAZ**
   - 名前タグ：後続手順書で指定（例：`<対象EC2名>-data`）
4. 「**ボリュームを作成**」をクリックします．
5. 作成したボリュームを選択 →「アクション ▼」→「**ボリュームをアタッチ**」
6. インスタンス：対象EC2を選択 → デバイス名：`/dev/sdf`（または推奨値）→「**ボリュームをアタッチ**」

#### 4.12.2 OS側でのフォーマットとマウント【実施対象：対象EC2】

操作手順：

1. 対象EC2にSSH接続します．
2. ブロックデバイス一覧を確認します．

```bash
lsblk
```

期待される出力例：

```text
NAME          MAJ:MIN RM  SIZE RO TYPE MOUNTPOINT
nvme0n1       259:0    0    8G  0 disk
└─nvme0n1p1   259:1    0    8G  0 part /
nvme1n1       259:2    0   50G  0 disk             ← 追加したボリューム
```

3. ファイルシステム（xfs）を作成します．

```bash
sudo mkfs -t xfs /dev/nvme1n1
```

4. マウントポイントを作成します．

```bash
sudo mkdir -p <マウントポイント>
```

（例：PostgreSQLの場合は `/data`）

5. 一時的にマウントして動作確認します．

```bash
sudo mount /dev/nvme1n1 <マウントポイント>
df -hT <マウントポイント>
```

6. 永続マウントのため `/etc/fstab` に登録します．まずUUIDを取得します．

```bash
sudo blkid /dev/nvme1n1
```

期待される出力例：

```text
/dev/nvme1n1: UUID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx" TYPE="xfs"
```

7. `/etc/fstab` に追記します．

ファイル編集：`/etc/fstab`

【追記内容】

```text
UUID=<取得したUUID>  <マウントポイント>  xfs  defaults,nofail  0  2
```

【理由】

OS再起動後も自動的にマウントされるようにするためです．

`nofail` を指定することで，万一ボリュームが見つからない場合でもOS起動を継続できます．

8. `/etc/fstab` の記述を検証します．

```bash
sudo mount -a
df -hT <マウントポイント>
```

エラーが出ないことを確認します．

注意点・コメント：

デバイス名は環境によって `/dev/xvdf` や `/dev/nvme1n1` などに変わります．`lsblk` で確認したものを使用してください．

`/etc/fstab` の記述ミスがあるとOSが起動しなくなる可能性があります．`mount -a` でエラーがないことを必ず確認してから再起動してください．

## 6. トラブルシューティング

### エラー1：SSH接続が「Connection timed out」になる

原因：

セキュリティグループのインバウンドルールで，作業端末のIPからのSSHが許可されていない可能性があります．

解決方法：

```bash
curl -s ifconfig.me
```

で作業端末のグローバルIPを再確認し，SGのインバウンドルールを更新します．

注意点・コメント：

自宅・オフィスのIPが動的に変わる場合は，IPが変わるたびにSGの更新が必要です．

------------------------------

### エラー2：SSH接続が「Permission denied (publickey)」になる

原因：

* キーペアの `.pem` ファイルのパーミッションが緩い
* 接続ユーザー名が間違っている

解決方法：

```bash
chmod 600 <キーペア名>.pem
ssh -i <キーペア名>.pem ec2-user@<踏み台サーバーのIP>
```

注意点・コメント：

Amazon Linux 2023 のデフォルトユーザーは `ec2-user` です．Ubuntu の場合は `ubuntu` になります．

------------------------------

### エラー3：Privateサブネット内のEC2から `dnf update` が失敗する

原因：

* NATゲートウェイが作成されていない
* Private用ルートテーブルに `0.0.0.0/0 → NATGW` がない
* PrivateサブネットがPrivate用ルートテーブルに関連付けられていない

解決方法：

VPC画面 →「ルートテーブル」→ Private用ルートテーブル →「ルート」タブ で `0.0.0.0/0` の経路を確認します．

「サブネットの関連付け」タブ で対象のPrivateサブネットが関連付けられているか確認します．

------------------------------

### エラー4：EBS追加ボリュームのマウント後，OSが再起動できない

原因：

`/etc/fstab` の記述ミスの可能性があります．

解決方法：

EC2を停止 → ルートボリュームをデタッチ → 別のEC2にアタッチ → `/etc/fstab` の該当行を修正．

注意点・コメント：

`nofail` オプションを `/etc/fstab` に必ず追加することで，このリスクを大幅に下げられます．

------------------------------

### エラー5：NATゲートウェイ作成時に「Elastic IPの上限に達しました」と表示される

原因：

リージョンあたりのEIP上限（デフォルト5個）に達しています．

解決方法：

不要なEIPを開放するか，AWSサポートに上限緩和申請を行います．

## 8. ロールバック手順

### 8.1 ロールバック判定基準

以下のいずれかが発生した場合は，ロールバックを検討します．

* 構築途中で大幅な設計変更が必要になった
* VPC CIDRを変更する必要が出てきた
* 検証完了後に環境を全削除したい

------------------------------

### 8.2 ロールバック順序

実施内容：依存関係の **逆順** に削除します．

削除順序：

```text
1. EC2インスタンス（踏み台 + 後続手順書で作成したもの全て）を終了
2. EBSボリューム（追加分）を削除
3. NATゲートウェイを削除
4. Elastic IPを開放
5. サブネットを削除
6. ルートテーブルを削除（メインルートテーブル以外）
7. IGWをVPCからデタッチ → 削除
8. セキュリティグループを削除（デフォルトSG以外）
9. VPCを削除
```

注意点・コメント：

NATGWは削除後もEIPは課金が継続します．**必ずEIPを別途開放してください**．

ロールバック後の確認：

* [ ] VPC画面で対象VPCが存在しないこと
* [ ] EC2画面で対象EC2が存在しないこと
* [ ] EIP画面で関連EIPが存在しないこと

## 付録B：設定ファイル解説

### B.1 `/etc/fstab`（EBS追加ボリュームの永続マウント）

OS起動時に自動マウントするファイルシステムを定義するファイルです．

```text
UUID=<取得したUUID>  <マウントポイント>  xfs  defaults,nofail  0  2
```

| 項目         | 内容                                                                 |
| ---------- | ------------------------------------------------------------------ |
| UUID       | ファイルシステムの一意識別子．`blkid` で取得                                       |
| マウントポイント   | OS側のマウント先ディレクトリ                                                    |
| ファイルシステム種別 | `xfs` / `ext4` など                                                  |
| マウントオプション  | `defaults,nofail` 推奨．`nofail` でブートを止めない                            |
| dump       | バックアップ要否．通常 `0`                                                    |
| fsck pass  | OS起動時のfsck順序．ルートは `1`，その他データボリュームは `2`                            |

### D.2 サブネットマスクの考え方

VPC `<VPC CIDR>` を4つのサブネットに分割する場合，ホスト部から2ビット分をネットワーク部に追加します．

| VPC CIDR | 4分割時のサブネット例                  |
| -------- | ---------------------------- |
| `/16`    | `/18` × 4（各16,382ホスト）        |
| `/16`    | `/20` × 16から4つ選択（各4,094ホスト） |

本手順書のデフォルト設計では `/20` × 4 を採用しています．

### D.3 NATゲートウェイのコスト最適化

NATゲートウェイは1時間ごとに課金されます．検証環境では以下の方法でコスト削減が可能です．

* 検証終了後はNATGWを削除し，再開時に再作成する
* 開発初期はNATGWを使わず，PrivateサブネットにもパブリックIPを一時的に付与する
