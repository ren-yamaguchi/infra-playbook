## カテゴリ F：ネットワーク・セキュリティ

### F-1. VPC ピアリング / Transit Gateway 模擬構成

2 つの VPC（異なる CIDR）間でピアリングを張り、EC2 同士がプライベート IP で通信できる構成を作る。マルチ VPC 設計の基礎となる重要テーマ。

**構成例**
- VPC-A（CIDR: 10.0.0.0/16）と VPC-B（CIDR: 10.1.0.0/16）をピアリング接続
- 各 VPC に EC2（t2.micro）を 1 台ずつ配置し、相互に ping / SSH が通ることを確認
- ルートテーブルの設定内容をドキュメント化する

**学習ポイント**
- VPC ピアリングは推移的ルーティング不可（A↔B、B↔C でも A と C は直接通信できない）
- Transit Gateway を使うと複数 VPC のハブ接続が可能（有料のため概念理解のみ推奨）

---

### F-2. NAT ゲートウェイ / NAT インスタンス

プライベートサブネットの EC2 からインターネットへのアウトバウンド通信を NAT 経由で実現する。

**構成例**
- パブリックサブネット：NAT インスタンス（t2.micro / 無料）または NAT ゲートウェイ
- プライベートサブネット：EC2（t2.micro）→ NAT 経由で yum update が通ることを確認

**学習ポイント**
- NAT インスタンスは無料枠で代替可能（`Source/Destination Check` を無効化する設定が必要）
- NAT ゲートウェイは時間課金・データ転送課金あり → 検証後すぐ削除でコスト管理

---

### F-3. 踏み台サーバー（Bastion Host）経由の多段 SSH

パブリックサブネットに Bastion を 1 台置き、プライベートサブネットの EC2 へ多段 SSH する構成。現場で最も頻繁に使うアクセス設計パターン。

**構成例**
- Bastion（パブリック）: SSH 許可 SG（自分の IP /32 のみ）
- ターゲット EC2（プライベート）: SSH 許可 SG（Bastion のプライベート IP /32 のみ）
- `~/.ssh/config` の `ProxyJump` 設定でワンコマンド接続を実現する

**学習ポイント**
- `ssh-agent` を使った鍵の転送（ForwardAgent）の仕組みを理解する
- SSM Session Manager（カテゴリ J-5）と比較し、使い分けを整理する

---

### F-4. iptables / nftables によるパケットフィルタリング

Security Group ではなく OS レベルでのファイアウォール設定。iptables の ACCEPT / DROP / FORWARD チェーンと NAT テーブルを操作する。

**課題内容**
- EC2 上で iptables ルールを手動作成し、特定ポート・特定 IP のみ許可する設定を行う
- FORWARD チェーンを利用して EC2 をソフトウェアルーターとして機能させる
- `iptables-save` / `iptables-restore` で設定を永続化する

**学習ポイント**
- AL2023 では nftables がデフォルト。`iptables-nft` による互換レイヤーの動作も確認する
- Security Group との二重管理になるため、どちらで制御すべきかの設計判断を意識する

---

### F-5. OpenVPN による VPN サーバー構築

EC2 上に OpenVPN サーバーを建て、クライアントから VPN 接続してプライベートリソースにアクセスする構成。

**課題内容**
- EC2（t2.micro）に OpenVPN をインストールし、PKI 証明書（Easy-RSA）を作成する
- クライアント（PC / スマートフォン）から VPN 接続し、プライベートサブネットの EC2 に到達できることを確認する

**学習ポイント**
- Split Tunneling（VPN 経由のトラフィックを選択的にルーティング）の設定
- AWS Client VPN（有料）との機能比較

## カテゴリ G：データベース発展

### G-1. PostgreSQL によるデータベースサーバー

MariaDB / MySQL ではなく PostgreSQL を EC2 上に構築し、ストリーミングレプリケーションを設定する。

**構成例**
- EC2-primary（PostgreSQL Primary）+ EC2-replica（PostgreSQL Replica）の 2 台構成
- `pg_basebackup` を使った初期同期、`recovery.conf` / `postgresql.conf` の設定
- レプリカラグを `pg_stat_replication` で確認する

**学習ポイント**
- MariaDB との設定ファイル構造・レプリケーション方式の違いを比較する
- WAL（Write Ahead Log）の概念を理解する

---

### G-2. Redis によるキャッシュサーバー

EC2 上に Redis を構築し、WordPress や Web アプリのセッション / オブジェクトキャッシュとして機能させる。

**構成例**
- EC2（t2.micro）に Redis をインストールし、WordPress の W3 Total Cache または Redis Object Cache プラグインと連携する
- `redis-cli` で `SET` / `GET` / `TTL` の基本操作と `INFO` によるステータス確認を行う

**学習ポイント**
- ElastiCache（有料）との機能比較と自己管理コストのトレードオフを整理する
- キャッシュ戦略（Cache-Aside / Write-Through / Write-Behind）の違いを理解する

---

### G-3. RDS（MySQL / PostgreSQL）との Web サーバー連携

RDS 無料枠（db.t3.micro / 20GB / 750h/月）を使い、EC2 上の Web サーバーから RDS に接続する構成を作る。

**構成例**
- EC2（Web + AP）→ RDS MySQL（シングル AZ）で WordPress を動作させる
- Security Group で EC2 のプライベート IP から 3306 番ポートのみ許可する設定
- RDS の自動バックアップ（スナップショット）を有効化し、ポイントインタイムリカバリを確認する

**学習ポイント**
- マルチ AZ の概念とフェイルオーバー動作を理解する（有効化は課金対象のため理解のみ推奨）
- RDS Parameter Group のカスタマイズ（文字コード設定など）を行う

---

### G-4. バックアップ自動化（mysqldump + S3 連携）

mysqldump / pg_dump の結果を cron で定期実行し、AWS CLI 経由で S3 バケットに自動アップロードする。

**課題内容**

```bash
# バックアップスクリプト例（/usr/local/bin/db_backup.sh）
#!/bin/bash
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="/tmp/backup_${DATE}.sql.gz"
mysqldump -u root -p'password' --all-databases | gzip > "$BACKUP_FILE"
aws s3 cp "$BACKUP_FILE" s3://your-backup-bucket/db/
# 7世代以上古いファイルを削除
aws s3 ls s3://your-backup-bucket/db/ | sort | head -n -7 | awk '{print $4}' | \
  xargs -I {} aws s3 rm s3://your-backup-bucket/db/{}
```

**学習ポイント**
- IAM ロール（インスタンスプロファイル）で AWS CLI 認証を行い、アクセスキーをコードに書かない設計にする
- S3 ライフサイクルポリシーによる自動削除との使い分けを理解する

## カテゴリ H：ストレージ・ファイル共有

### H-1. EFS（Elastic File System）による共有ストレージ

複数の EC2 から同一の EFS をマウントし、NFS と比較した際の管理コスト・スケーラビリティの違いを体験する。

**構成例**
- EFS ファイルシステムを作成し、2 台の EC2 から同時にマウントする
- 一方の EC2 で書き込んだファイルが即座にもう一方から読み取れることを確認する
- WordPress の `/var/www/html` を EFS に配置し、ALB 配下の複数 EC2 でコンテンツを共有する

**学習ポイント**
- EFS は GB 単位の従量課金（無料枠：5GB / 月）。大量データの書き込みに注意する
- マウントオプション（`nfsvers=4.1`、`tls`）とパフォーマンスモードの選択基準を理解する

---

### H-2. S3 マウント（Mountpoint for Amazon S3）

S3 バケットを EC2 上にファイルシステムとしてマウントし、静的コンテンツ配信や大容量データの読み書きに活用する。

**課題内容**
- `mount-s3` コマンドで S3 バケットを `/mnt/s3` にマウントする
- WordPress のメディアファイル（`wp-content/uploads`）を S3 にマウントして提供する構成を試す
- 書き込み制限（シーケンシャル書き込みのみ対応）の動作を確認する

---

### H-3. rsync + cron による定期バックアップ

2 台の EC2 間で rsync を使い、差分バックアップを定期実行する。SSH 鍵認証と組み合わせた無人自動化。

**課題内容**

```bash
# /usr/local/bin/rsync_backup.sh
#!/bin/bash
rsync -avz --delete \
  -e "ssh -i /home/ec2-user/.ssh/backup_key" \
  /var/www/html/ \
  ec2-user@10.0.1.100:/backup/html/
```

**学習ポイント**
- `--delete` オプションの挙動（ソース側で削除されたファイルはターゲット側でも削除）に注意する
- `rsync` の差分転送アルゴリズムによる通信量削減の仕組みを理解する

---

### H-4. LVM（Logical Volume Manager）による EBS ボリューム管理

EC2 に EBS を追加アタッチし、LVM で論理ボリュームを構成する。本番環境でのディスク容量管理の基礎。

**課題内容**
- EBS ボリューム（gp3 / 8GB 以内で無料枠）を EC2 に追加アタッチする
- `pvcreate` → `vgcreate` → `lvcreate` → `mkfs` → `mount` の流れで論理ボリュームを作成する
- `lvextend` + `resize2fs` で稼働中のボリュームをオンライン拡張する

**学習ポイント**
- 無料枠の EBS 合計は 30GB（gp2 / gp3）。複数ボリュームの合計が超えないよう管理する
- EBS スナップショットとの組み合わせによるバックアップ戦略を設計する

## カテゴリ I：コンテナ・オーケストレーション

### I-1. Docker Compose によるマルチコンテナ構成

EC2 上で Nginx + PHP-FPM + MariaDB を Docker Compose で構築する。手動構築との設定の対比を意識する。

**構成例（docker-compose.yml 概要）**

```yaml
services:
  nginx:
    image: nginx:alpine
    ports: ["80:80"]
    volumes: ["./nginx.conf:/etc/nginx/conf.d/default.conf"]
  php-fpm:
    build: ./php
    volumes: ["./html:/var/www/html"]
  mariadb:
    image: mariadb:10.11
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: wordpress
```

**学習ポイント**
- `docker network`・`volume`・`depends_on`・`healthcheck` の使い方を習得する
- EC2 上のコンテナ構成と手動構築の設定量・再現性を比較する

---

### I-2. ECR + EC2 によるコンテナデプロイパイプライン

カスタム Docker イメージを ECR にプッシュし、EC2 で pull して起動する基本的なデプロイフローを体験する。

**フロー**
1. ローカルで `docker build` → カスタムイメージ作成
2. `aws ecr get-login-password` で ECR 認証
3. `docker tag` → `docker push` で ECR にプッシュ
4. EC2 上で `docker pull` → `docker run` で起動

**学習ポイント**
- ECR の無料枠：プライベートリポジトリ 500MB / 月。イメージの整理を習慣化する
- IAM ロールで EC2 に ECR 読み取り権限（`AmazonEC2ContainerRegistryReadOnly`）を付与する

---

### I-3. ECS（Fargate）によるコンテナ実行

ECS Fargate で静的コンテンツを配信するコンテナを動かす最小構成を体験する。

**注意事項**
- Fargate は無料枠対象外（vCPU・メモリの時間課金）
- 最小構成（0.25vCPU / 0.5GB）で短時間の検証後すぐ削除することでコストを最小化する

**構成例**
- ECR に Nginx イメージをプッシュし、ECS クラスター → タスク定義 → サービスの順で起動
- ALB とターゲットグループを紐付けてアクセスを確認する

**学習ポイント**
- タスク定義（CPU / メモリ / コンテナ定義）と EC2 起動タイプの概念的な違いを理解する
- Fargate vs EC2 起動タイプの選択基準をまとめる

---

### I-4. Docker ネットワーク / ブリッジ設計

`bridge`・`host`・`none` の各ネットワークモードの動作を EC2 上で検証する。

**課題内容**
- `bridge`モード：デフォルトのコンテナネットワーク。NAT 経由でホスト外と通信
- `host`モード：コンテナがホストのネットワークスタックを共有。ポートマッピング不要
- カスタムブリッジネットワークを作成し、コンテナ名での DNS 解決を確認する

**学習ポイント**
- カテゴリ①〜⑧で構築した環境をコンテナ化する際のネットワーク設計に直結する知識

## カテゴリ J：CI/CD・自動化

### J-1. GitHub Actions による EC2 への自動デプロイ

GitHub へのプッシュをトリガーに EC2 へ自動デプロイするパイプラインを構築する。現場での採用率が高い。

**パイプライン例（.github/workflows/deploy.yml 概要）**

```yaml
name: Deploy to EC2
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: SSH Deploy
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.EC2_HOST }}
          username: ec2-user
          key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            cd /var/www/html
            git pull origin main
            sudo systemctl reload httpd
```

**学習ポイント**
- Secrets（`EC2_HOST`、`EC2_SSH_KEY`）の管理方法を理解する
- デプロイ失敗時のロールバック戦略を考慮する

### J-5. AWS Systems Manager（SSM）による EC2 管理

SSH キーなしで SSM Session Manager から EC2 に接続する。キー管理コストとセキュリティを改善する。

**設定手順**

**追加機能（課題内容）**

- Patch Manager で OS パッチを自動適用するメンテナンスウィンドウを設定する

## カテゴリ K：監視・ログ・可観測性

### K-1. CloudWatch エージェントによるカスタムメトリクス収集

3. CloudWatch ダッシュボードでカスタムメトリクスを可視化する
4. メモリ使用率 80% 超でアラームを作成し、SNS でメール通知する

**学習ポイント**
- エージェントなしで取得できる標準メトリクス（CPU / ネットワーク / ディスク I/O）との違いを整理する

---

### K-2. CloudWatch Logs Insights によるログ分析

Apache / Nginx のアクセスログを CloudWatch Logs に転送し、クエリで分析する。

**クエリ例**

```sql
-- HTTP ステータスコード別のリクエスト数を集計
fields @timestamp, @message
| parse @message '* * * * * "* *" * *' as host, ident, user, time, tz, method, request, status, size
| stats count(*) by status
| sort count desc
```

**学習ポイント**
- ログの保持期間設定（デフォルト無期限は課金増大の原因）を適切に設定する
- Logs Insights のクエリ構文（`parse`・`filter`・`stats`・`sort`）を習得する

---

### K-3. Prometheus + Grafana による可視化ダッシュボード

EC2 上に Prometheus と Grafana を構築し、Node Exporter で収集したシステムメトリクスを可視化する。

**構成例**
- EC2-monitor：Prometheus + Grafana
- EC2-target × 2：Node Exporter（ポート 9100）
- Grafana のダッシュボードに「Node Exporter Full」（Dashboard ID: 1860）をインポートする

**学習ポイント**
- Prometheus のスクレイプ設定（`prometheus.yml`）とラベルの扱い方を理解する
- Zabbix と Prometheus の設計思想の違い（pull 型 vs push 型）を比較する

### K-4. Fluent Bit によるログ集約パイプライン

各 EC2 の Fluent Bit がログを収集し、CloudWatch Logs または S3 に転送するログパイプラインを構築する。

**構成例（fluent-bit.conf 抜粋）**

```ini
[INPUT]
    Name  tail
    Path  /var/log/httpd/access_log
    Tag   apache.access

[OUTPUT]
    Name  cloudwatch_logs
    Match apache.*
    region ap-northeast-1
    log_group_name /ec2/apache/access
    log_stream_prefix instance-
```

**学習ポイント**
- Fluentd（Ruby ベース、機能豊富）と Fluent Bit（C ベース、軽量）の使い分けを理解する
- フィルタープラグイン（`grep`・`record_modifier`）でログを整形してから転送する

### K-5. AWS CloudTrail による操作ログ記録

- CloudWatch Logs と連携させ、ルートアカウントのログイン時にアラームを発火させる

**学習ポイント**

## カテゴリ L：セキュリティ強化

### L-1. AWS WAF の導入

ALB に AWS WAF を紐付け、代表的な攻撃パターンをマネージドルールでブロックする。

**課題内容**

### L-2. fail2ban による SSH ブルートフォース対策

fail2ban を EC2 に導入し、SSH への不正ログイン試行を自動的に遮断する。

**設定例（/etc/fail2ban/jail.local）**

[sshd]

### L-3. ClamAV によるウイルススキャン自動化

ClamAV をインストールし、cron で定期的にウイルススキャンを実行・記録する。

**課題内容**

```bash
# /etc/cron.daily/clamscan
#!/bin/bash
LOGFILE="/var/log/clamav/scan_$(date +%Y%m%d).log"
freshclam --quiet
clamscan -r /var/www/html --log="$LOGFILE" --quiet
# 感染ファイルが見つかった場合にメール通知
if grep -q "FOUND" "$LOGFILE"; then
    mail -s "[ALERT] ClamAV Virus Found" admin@example.com < "$LOGFILE"
fi
```

**学習ポイント**
- `freshclam` によるウイルス定義データベースの自動更新を設定する
- スキャン対象ディレクトリの選定（全体スキャン vs 特定ディレクトリ）とパフォーマンスへの影響を考慮する

### L-4. SELinux によるプロセス制限

SELinux（AL2023 のデフォルト有効）の動作を理解し、ポリシーを正しく設定する。

**課題内容**
- `getenforce` で現在のモードを確認し、`enforcing` / `permissive` の切り替えを体験する
- Apache が非標準ポート（例: 8080）でリッスンできない理由を SELinux の観点から調査し、`semanage port` で解決する
- `ausearch` / `audit2allow` を使ってアクセス拒否ログからカスタムポリシーを作成する

**学習ポイント**
- SELinux の無効化（`disabled`）は本番禁止。`permissive` で調査 → `enforcing` で適用が正しい手順

### L-5. AWS Secrets Manager / Parameter Store の活用

DB パスワードや API キーをコードにハードコードせず、AWS のシークレット管理サービスから取得する。

**構成例（Python / boto3）**

```python
import boto3, json

def get_db_secret():
    client = boto3.client('secretsmanager', region_name='ap-northeast-1')
    response = client.get_secret_value(SecretId='prod/myapp/db')
    return json.loads(response['SecretString'])

secret = get_db_secret()
conn = connect(host=secret['host'], user=secret['username'], password=secret['password'])
```

**学習ポイント**
- Secrets Manager（自動ローテーション機能あり / 有料）と SSM Parameter Store（無料枠あり）の使い分け
- IAM ポリシーでアクセスできるシークレットを最小権限に絞る設計

## カテゴリ M：高可用性・スケーリング

### M-1. Route 53 ヘルスチェックとフェイルオーバールーティング

Route 53 のヘルスチェックで Primary EC2 の死活を監視し、障害時に Secondary EC2 へ自動切り替えする。

**構成例**
- Primary レコード（EC2-a）: フェイルオーバーポリシー PRIMARY + ヘルスチェック設定
- Secondary レコード（EC2-b）: フェイルオーバーポリシー SECONDARY
- Primary を停止してフェイルオーバーが発生することを確認する

**学習ポイント**
- ヘルスチェックの確認間隔（10秒 or 30秒）と閾値の設定
- Route 53 ヘルスチェックは $0.50/チェック/月。検証後は削除する

---

### M-2. Auto Scaling Group + ALB による動的スケーリング

CPU 使用率に応じて EC2 台数が自動増減する構成を ALB と組み合わせて構築する。

**構成例**
- 起動テンプレートに User Data でミドルウェアの自動インストールを記述する
- スケーリングポリシー：CPU 使用率 70% 超で +1 台、30% 未満で -1 台
- `stress` コマンドで CPU 負荷をかけてスケールアウトを確認し、停止後にスケールインを確認する

**学習ポイント**
- スケールイン保護（Scale-In Protection）を設定して必要台数を維持する
- 台数増加によるコスト増大に注意。検証後は最大台数を 0 に設定して EC2 を停止する

---

### M-3. EC2 スナップショットと AMI によるバックアップ / 復旧

動作中の EC2 から AMI を作成し、別インスタンスとして起動できることを確認する。

**課題内容**
- WordPress が動作する EC2 から AMI を作成する（`aws ec2 create-image`）
- 作成した AMI から新しい EC2 を起動し、WordPress がそのまま動作することを確認する
- AWS Backup でバックアッププランを作成し、定期スナップショットを自動化する

**学習ポイント**
- AMI 作成時の再起動オプション（`--no-reboot`）と整合性の関係を理解する
- EBS スナップショットは S3 に保存される。ストレージ料金（$0.05/GB/月）に注意する

### M-4. マルチ AZ 構成による耐障害性設計

**学習ポイント**

### N-2. Varnish Cache によるリバースプロキシキャッシュ

Apache / Nginx の前段に Varnish を置き、静的コンテンツをキャッシュしてバックエンド負荷を削減する。

**構成例**
- EC2 上に Varnish（ポート 80）+ Apache（ポート 8080）を同居させる
- VCL（Varnish Configuration Language）で画像・CSS・JS のキャッシュ TTL を設定する
- `varnishstat` / `varnishlog` でキャッシュヒット率を確認する

**学習ポイント**
- CloudFront（CDN）との役割の違い：Varnish はオリジンサーバー近傍のキャッシュ、CloudFront はエッジキャッシュ

### N-3. HAProxy によるロードバランサー

ALB を使わず EC2 上の HAProxy でレイヤー 4 / レイヤー 7 の負荷分散を実装する。

**設定例（/etc/haproxy/haproxy.cfg 抜粋）**

```
frontend http_front
    bind *:80
    default_backend http_back

backend http_back
    balance roundrobin
    option httpchk GET /health
    server web1 10.0.1.10:80 check
    server web2 10.0.1.11:80 check
```

**学習ポイント**
- HAProxy の統計ページ（`stats uri /haproxy?stats`）で各バックエンドの状態をリアルタイム確認する
- ALB との比較：HAProxy はより低レイテンシで細かい制御が可能だが自己管理コストが高い

### N-4. Let's Encrypt ワイルドカード証明書の自動更新

Certbot と DNS-01 チャレンジを使い、`*.example.com` のワイルドカード証明書を取得・自動更新する。

**手順概要**
1. Route 53 にドメインを登録（または既存ドメインの NS を変更）
2. `certbot certonly --dns-route53 -d "*.example.com" -d "example.com"` で取得
3. `cron` または `systemd timer` で `certbot renew` を定期実行する

**学習ポイント**
- DNS-01 チャレンジは HTTP アクセスが不要なため、プライベートサブネットの EC2 でも取得可能
- 証明書の有効期限（90 日）と自動更新のタイミング（30 日前）を管理する

AWS CLI または boto3（Python）を使い、日常運用タスクを自動化するスクリプトを作成する。

**課題例**

**学習ポイント**
- boto3 のクライアント vs リソースの使い分けを理解する
- IAM ロールで EC2 に必要最小限の権限を付与し、アクセスキーを使わない設計にする
