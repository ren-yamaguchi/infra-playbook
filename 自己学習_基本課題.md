## 1. Apache によるシングル構成 Web サーバー

### 復習課題

- EC2（AL2023）で Apache を構築し、バーチャルホストで 2 つのドメインを同一 IP で振り分ける
- `.htaccess` でアクセス制限（特定 IP のみ許可）とリダイレクト（HTTP → HTTPS）を設定する

### 応用・発展課題

- Let's Encrypt（Certbot）で SSL/TLS 証明書を取得し、HTTPS 化する
- Apache のアクセスログを AWS CloudWatch Logs にエージェント経由で転送・可視化する

---

## 2. Apache + Tomcat（Web / AP サーバー）

### 復習課題

- Apache + Tomcat を 2 台の EC2 に分離（Apache はリバースプロキシとして Tomcat に転送）
- `mod_jk` または `mod_proxy_ajp` での AJP 接続と `mod_proxy_http` での HTTP 接続を両方試す

### 応用・発展課題

- Tomcat 側に JDBC で接続するサンプルアプリを配置し、セッション情報を MariaDB に永続化する
- Apache で Tomcat 2 台へのラウンドロビン負荷分散を `mod_proxy_balancer` で構成する（ALB 不使用）

---

## 3. Apache + PHP-FPM + MariaDB（LAMP / WordPress）

### 復習課題

- 3 台 EC2 構成（Web / AP / DB 分離）をゼロから再構築し、Security Group の設定根拠をドキュメント化する
- MariaDB のスロークエリログを有効化し、`mysqldumpslow` で解析する

### 応用・発展課題

- MariaDB にレプリケーション（Primary / Replica）を追加し、WordPress の読み取りを Replica に向ける
- PHP-FPM のプロセス管理設定（`pm.max_children` 等）をチューニングし、`ab`（Apache Bench）で負荷テストを行う

---

## 4. Apache + PHP-FPM + MariaDB + BIND（DNS 名前解決）

### 復習課題

- フォワードゾーンとリバースゾーンを再設定し、`dig`・`nslookup` で正引き / 逆引きを確認する
- BIND のビュー機能（`view` ディレクティブ）で内部向けと外部向けの応答を分ける

### 応用・発展課題

- ゾーン転送（Primary / Secondary BIND）を 2 台間で構成し、フェイルオーバーを検証する
- Route 53 プライベートホストゾーンと BIND を比較検証し、それぞれの名前解決動作の違いをまとめる

---

## 5. Postfix + Dovecot + BIND（メールサーバー）

### 復習課題

- 2 台構成（MTA と MDA 分離）をゼロから再構築し、SMTP と IMAP/POP3 の通信フローを図示する
- Postfix で送受信ログを解析し、正常配送とエラーの判別をできるようにする

### 応用・発展課題

- Postfix に SMTP 認証（SASL）と TLS 暗号化を組み合わせて設定する
- SPF レコードを BIND のゾーンファイルに追加し、`spf-tools` で検証する

---

## 6. Postfix + Dovecot + BIND + NFS（メール共有環境）

### 復習課題

- NFS のエクスポート設定（`/etc/exports`）とマウント設定を再構築し、Maildir 共有を確認する
- NFS のマウントオプション（`hard` / `soft`、`timeo`）の違いを検証する

### 応用・発展課題

- NFS を EFS（Elastic File System）に置き換えてメールディレクトリを共有する（無料枠内で利用可能）
- Dovecot のクォータ機能でユーザーごとのメールボックス容量制限を設定する

---

## 7. Zabbix（監視環境）

### 復習課題

- Zabbix エージェントを追加の EC2 に導入し、CPU / メモリ / ディスク使用率のトリガーを設定する
- アクション設定でトリガー発火時にメール通知を送るよう設定し、テスト発火で確認する

### 応用・発展課題

- Zabbix の LLD（Low Level Discovery）でネットワークインターフェースやファイルシステムを自動検出する
- CloudWatch カスタムメトリクスと Zabbix を並行運用し、監視項目・コスト・運用性を比較する

---

## 8. ALB + Apache（負荷分散）

### 復習課題

- ALB のターゲットグループに 3 台目の EC2 を追加し、スティッキーセッション（Cookie）の ON/OFF で振る舞いの違いを確認する
- ALB のアクセスログを S3 に出力し、ログ内容を確認する

### 応用・発展課題

- ALB リスナールールでパスベースルーティングを設定する（例：`/api/*` → 別ターゲットグループ）
- Auto Scaling Group と連携させ、CPU 負荷に応じて EC2 台数が増減する構成を作る（スケールイン後すぐ削除でコスト管理）

### Ansible

演習では EC2 複数台を使い、Ansible の基本モジュール（ping / dnf / systemd / copy / user / lineinfile / script）から始まり、shell / command モジュール禁止の制約下で Apache + Tomcat 環境を Playbook で完全構築した。

#### 復習課題

| # | 課題 | 使用モジュール | 確認ポイント |
|---|------|--------------|------------|
| 1 | 2 台の EC2 で互いに ping が通る双方向 Ansible 環境を再構築する | `ping` | `ansible -i hosts -m ping <IP>` が双方向で成功すること |
| 2 | Playbook で相手サーバに Apache をインストールする | `dnf` | `state: present` と `state: absent` の冪等性を確認する |
| 3 | Playbook で Apache を起動・停止・再起動する | `systemd` | `enabled: yes` と `state: started/stopped` の違いを確認する |
| 4 | 自サーバの `/var/tmp/index.html` を相手の `/var/www/html/` にコピーし、グローバル IP でアクセスして表示を確認する | `copy` | ファイルのパーミッション（`mode`）も合わせて指定する |
| 5 | 相手サーバに任意のユーザーを作成し、`id` コマンドで存在を確認する | `user` | `state: absent` で削除も試し、冪等性を確認する |
| 6 | 相手サーバの `/var/www/html/index.html` 内の文字列を置換する | `lineinfile` | `regexp` と `line` の組み合わせを変えて複数パターンを試す |
| 7 | 自サーバで作成したシェルスクリプトを相手サーバで実行する | `script` | スクリプトの実行結果を `register` で受け取り `debug` で出力する |
| 8 | shell / command モジュール禁止の制約下で Tomcat を構築し、ブラウザで猫を表示する | `get_url` `unarchive` `template` `systemd` | 一から Playbook を書き直し、タスクの依存関係（`notify` / `handlers`）を整理する |
| 9 | Playbook で Apache を構築し Tomcat と連携させ、グローバル IP で猫を表示する | `template` `systemd` | `mod_proxy_ajp` の設定ファイルを `template` モジュールで配置する |
| 10 | EC2 を全削除して再構築し、課題 8・9 の Playbook を再適用して完全再現できることを確認する | 全モジュール | 再実行しても全タスクが `ok` または `changed` で完了し `failed` が出ないことを確認する |

#### 応用・発展課題

- **Playbook のロール化**：課題 8・9 の Playbook を `roles/` ディレクトリ構造に分割し（`tasks` / `handlers` / `templates` / `defaults`）、再利用可能なロールとして整理する
- **変数の外部化**：ポート番号・ユーザー名・ドメイン名などをすべて `vars/` または `group_vars/` に切り出し、Playbook 本体からハードコードを排除する
- **Ansible Vault によるシークレット管理**：DB パスワードや SSH 鍵などの機密情報を `ansible-vault encrypt` で暗号化し、安全に Playbook に組み込む
- **複数環境へのデプロイ**：`inventory/dev` と `inventory/stg` を作成し、同一 Playbook を環境ごとに切り替えて適用できる構成にする
- **Ansible + Terraform の連携**：Terraform で EC2 を起動し、起動後のプロビジョニングを Ansible が担当するパイプラインを構築する（IaC と構成管理の責任分界を体感する）

### Docker

演習では AmazonLinux2023 の EC2 上に Docker / Docker Compose をインストールし、既存の WordPress イメージを使わず AmazonLinux2 ベースのカスタムイメージで web/ap コンテナと db コンテナを分離構築した。ボリュームによるデータ永続化（コンテナ削除後も記事・設定が維持される）まで実装した。

#### 復習課題

| # | 課題 | 確認ポイント |
|---|------|------------|
| 1 | AmazonLinux2023 の EC2 に Docker と Docker Compose を再インストールし、`docker info` と `docker compose version` で動作を確認する | インストール手順をスクリプト化しておく |
| 2 | AmazonLinux2 ベースのカスタム WordPress イメージ（web/ap）を Dockerfile から再ビルドし、`docker images` で確認する | `FROM amazonlinux:2` でベースを固定し、Apache + PHP + WordPress のインストールをレイヤー最小化を意識して記述する |
| 3 | web/ap コンテナと db コンテナを分離した `docker-compose.yml` を再作成し、`docker compose up -d` で WordPress が起動することを確認する | コンテナ間通信に `networks` を使い、db コンテナのポートをホストに公開しない設定にする |
| 4 | WordPress のデータ永続化ディレクトリを特定し（`/var/www/html/wp-content` と MariaDB データディレクトリ）、`docker compose down && docker compose up -d` 後も投稿記事が残ることを確認する | `volumes:` の named volume と bind mount の違いを試す |
| 5 | コンテナのログを `docker compose logs -f` で確認し、Apache エラーログと MariaDB ログをそれぞれ読み取る | ログドライバー（`json-file`）のオプション（`max-size` / `max-file`）を設定してログローテーションを実装する |

#### 応用・発展課題

- **マルチステージビルドによるイメージ軽量化**：WordPress カスタムイメージを builder ステージと runtime ステージに分割し、最終イメージのサイズを削減する
- **ECR へのカスタムイメージのプッシュと EC2 での pull 起動**：ビルドしたカスタムイメージを ECR にプッシュし、別の EC2 で pull して同じ WordPress 環境を再現する
- **docker-compose.yml の環境変数外部化**：DB パスワード・ユーザー名などを `.env` ファイルに切り出し、`docker-compose.yml` 本体からハードコードを排除する
- **ヘルスチェックの実装**：`docker-compose.yml` の `healthcheck` で web/ap・db コンテナの死活を監視し、db が Ready になるまで web/ap が起動を待つ依存関係を制御する
- **Nginx リバースプロキシコンテナの追加**：web/ap コンテナの前段に Nginx コンテナを追加し、SSL ターミネーションとリバースプロキシを担当する 3 コンテナ構成に発展させる
- **Ansible との連携**：Docker / Docker Compose のインストールから `docker compose up` までを Ansible Playbook で自動化し、EC2 を新規起動するだけで WordPress 環境が立ち上がる構成を実装する

## 10. 取り組み順の目安

```
Step 1：まず固める（復習）
  └─ ③④⑦⑧ の復習課題

Step 2：演習の復習と定着
  └─ Ansible 復習課題（課題 1〜10 の再実施・Playbook 整理）
  └─ Docker 復習課題（カスタムイメージ再ビルド・永続化の再確認）

Step 3：次に広げる（応用）
  └─ ③ レプリケーション
  └─ ⑧ Auto Scaling 連携
  └─ Ansible ロール化・Vault によるシークレット管理
  └─ Docker ECR 連携・Nginx リバースプロキシコンテナ追加

Step 4：並行して新規習得
  └─ C（Terraform × AWS）  ← 最優先
  └─ Ansible + Terraform 連携（IaC × 構成管理の組み合わせ）
  └─ D（S3 + CloudFront）

Step 5：余裕があれば
  └─ A（Nginx）
  └─ E（IAM 設計）
```

> **Terraform × AWS を最優先にする理由**  
> 手動構築の経験がそのままコードに変換できる今が、最も身につきやすいタイミングです。  
> VPC・SG・EC2 の作成を Terraform で再現することで、インフラ全体の構造理解も深まります。
>
> **Ansible・Docker 演習直後に復習を挟む理由**  
> 演習中は「動かすこと」が優先になりやすいため、直後に Playbook の構造整理・変数外部化・冪等性の確認を行うことで、現場で使えるレベルの理解に定着させることができます。
