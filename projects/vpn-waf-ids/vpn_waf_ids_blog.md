# 「攻撃をいくつもの壁で止める」── VPN・WAF・多層監視を自分で構築してわかったこと

---

## はじめに

「セキュリティ」と聞くと、なんとなく難しそう、専門家の領域、と感じる方も多いのではないでしょうか。

今回の学習では、**VPN・WAF・侵入検知**という、セキュリティの現場でよく耳にする3つの仕組みを、実際にAWS上に手を動かして構築しました。

この記事では、「何を作ったのか」「それぞれの仕組みが何をしているのか」を、エンジニア以外の方にも伝わるように書いてみます。難しいコマンドや設定値の話はなるべく省いて、「概念として何が面白いか」に絞っています。

---

## 何を作ったのか

今回構築したのは、**4台のサーバーが連携する、多層防御(Defense in Depth)の実験環境**です。

```
【インターネット(外の世界)】
        │
        ├─── WAFサーバー      … Web攻撃を検知・遮断する「門番」
        │        │
        │        └─── Webサーバー … 実際のコンテンツを返す「本体」
        │                             (WAFからしかアクセスできない)
        │
        └─── VPN-GWサーバー  … 暗号化トンネルで管理経路を確保する「鍵付きの通用口」
                 │
                 └─── VPN Clientサーバー … 安全なトンネルを通って中に入れる「許可証を持った人」
```

ポイントは「**1つの防御で全部守ろうとしない**」という設計思想です。仮に1つの壁を突破されても、次の壁が待ち構えています。これを多層防御といいます。

## 登場人物（ミドルウェア）の紹介

今回の構成で初めて触れたミドルウェアを、できるだけシンプルに紹介します。

### WireGuard ── 「暗号化された秘密の通路」

VPN（Virtual Private Network）を実現するソフトウェアです。

わかりやすくいうと、「インターネット上に、見えない専用トンネルを掘る」ものです。このトンネルを通った通信は暗号化されており、外部から盗聴できません。

設定の特徴として、**認証に「公開鍵」しか使わない**点が挙げられます。パスワードが不要なのでブルートフォース攻撃(総当たり)に強く、「鍵の登録がなければそもそも存在しないサーバのように振る舞う」という設計になっています。

> **豆知識:** WireGuard は2020年にLinuxカーネルへ正式統合されたモダンなVPNプロトコルです。歴史的な定番であるOpenVPNやIPsecと比べて設定がシンプルで、コードが数千行と小規模なため「読んで理解できるVPN」として注目されています。

---

### ModSecurity + OWASP CRS ── 「リクエストを検査する空港の手荷物検査機」

**ModSecurity**は、Webサーバー(Apache)に組み込まれるWAF(Web Application Firewall)エンジンです。WAFとは、HTTPリクエストの中身を解析して、攻撃パターンを含むリクエストを遮断するしくみです。

ただし、ModSecurity単体では「検査台」があるだけで、何が危険かを知りません。そこで**OWASP CRS(Core Rule Set)**を組み合わせます。CRSは「これが危険なリクエストのパターン一覧」という検査ルール集で、SQL注入やXSSなどの代表的な攻撃手法が数百件登録されています。

ModSecurity + CRS の組み合わせで、「空港の手荷物検査機 + 禁止物リスト」のように機能します。

**実際に試したこと:**

```
# SQLインジェクション攻撃のテスト(安全な検証用リクエスト)
curl -G "http://WAFサーバー/" --data-urlencode "id=1' OR '1'='1"
→ 403 Forbidden が返ってきた ✅

# XSS攻撃のテスト
curl -G "http://WAFサーバー/" --data-urlencode "q=<script>alert(1)</script>"
→ 403 Forbidden が返ってきた ✅
```

WAFが攻撃と判断したリクエストは、バックエンドのWebサーバーへは届きません。Webサーバーは「WAFが許可したものしか来ない」という前提で動けるため、役割がきれいに分離されています。

> **ポイント:** OWASP CRSのルールは「リクエストに`<script>`が含まれていたら怪しい」というパターンマッチで動きます。そのWebサーバーが本当にXSSの脆弱性を持っているかどうかは確認せず、「怪しいパターンは入口で弾く」という方針です。これは過剰検知(誤検知)を生む可能性もありますが、「アプリ側の実装を信用せず、入口で守る」というWAFの基本哲学の表れです。

### fail2ban ── 「鍵を間違えすぎた人を締め出す番人」

SSHログイン失敗のログを監視し、一定回数(今回は5回)失敗した送信元IPアドレスを自動的にブロックするツールです。

ブルートフォース攻撃(パスワードの総当たり)に対して有効で、「5回失敗したら10分間入り口を封鎖する」といった設定ができます。

面白いのは、**fail2banがログを見て、iptables(Linuxのファイアウォール)を動的に書き換える**という仕組みです。攻撃を検知したらリアルタイムでルールを追加し、BAN期間が終わったら自動でルールを削除します。

**実際に試したこと:**

VPNクライアントサーバーからVPN-GWサーバーへ、意図的にSSH認証を5回失敗させました。6回目の接続試行時点では、すでにそのIPアドレスがブロックされており、即座に接続が拒否されました。

---

### auditd + iptables LOG ── 「何が起きたかを記録する監視カメラ」

**auditd**はLinuxの監査デーモンで、「誰が、いつ、どのファイルを変更したか」をログに記録します。例えば`/etc/passwd`(ユーザーアカウント情報)や`/etc/wireguard/`(VPN設定)の変更を監視し、改ざんの痕跡を残します。

**iptables LOG**は、「許可していないUDPパケットが来た」「短時間に大量のSYNパケットが来た」といったネットワーク上の不審な動きをsyslogに記録します。これ自体は攻撃を止めるものではなく、「何があったかを記録する」ことに特化しています。

> **なぜSuricata(本格的なIDS)を使わなかったのか:**
> 当初はSuricata(侵入検知システム)の導入を検討していました。しかしAmazon Linux 2023の標準リポジトリで提供されておらず、ソースビルドにはt3.microのメモリでは不足するリスクがありました。
> 
> 調べていくと、本番クラウド環境ではSuricata相当のシグネチャ検知は**AWS GuardDuty**のようなマネージドサービスで代替し、ホスト上では**auditd**でファイル・プロセス操作を監視するという分担が現実的であることがわかりました。今回の構成はその現実の運用に近い形になっています。

## 「多層防御」を体感したポイント

構築を通じて、いくつか「なるほど」と感じた場面がありました。

### 「同じソフトウェアでも、役割は全く変わる」

今回、ApacheというWebサーバーソフトウェアを2台のサーバーで使いましたが、役割はまったく異なります。

- **WAFサーバーのApache**: 受け取ったリクエストをModSecurityで検査し、バックエンドへ転送する「関所」
- **WebサーバーのApache**: 静的HTMLをただ返す「コンテンツの棚」

同じソフトウェアでも、組み合わせるモジュールと配置によって全く違う機能を担います。「ソフトウェアの機能 = 一つの役割」ではないというインフラ設計の本質を体感しました。

### 「WAFをバイパスする経路を物理的に塞ぐ」

WAFを構築しても、攻撃者がWebサーバーへ直接アクセスできてしまえば意味がありません。そこで、**WebサーバーのセキュリティグループでHTTP(80番)へのアクセスをWAFのIPアドレスからのみ許可**しました。

これにより、仮に誰かがWAFを通らずにWebサーバーへアクセスしようとしても、パケットレベルで破棄されます。「WAFを設定する」だけでなく「WAFを通らない経路を塞ぐ」まで行って初めて、WAFが意味を持ちます。

### 「IPフォワードという概念」

Linuxは初期設定では「自分宛てのパケットしか処理しない」ようになっています。VPN-GWサーバーがVPNトンネルから届いたパケットをVPC内の他サーバーへ転送するためには、明示的に「IP転送」を有効化する必要がありました。これを知らないと、「VPNには繋がるのに、その先に届かない」という現象に直面します（実際に直面しました）。

### 「ログは誤検知との戦い」

iptables LOGを仕掛けて通信を観察していると、意図しないログが大量に記録されます。

例えば、AWS内部のNTPサーバー(時刻同期用サーバー)からのパケットが「許可していないUDP通信」として記録されたり、自分自身のSSH接続がSYNバーストとして記録されたりしました。

これは攻撃ではありません。しかし、ログ上では攻撃と区別がつきません。**本物のセキュリティ運用では、こうした誤検知(False Positive)を丁寧に除外するチューニング作業が大半を占める**、ということを体感しました。

## 構築していてハマったこと

### Amazon Linux 2023は「iptablesが入っていない」

Amazon Linux 2023はiptablesの後継である`nftables`をデフォルトで採用しています。そのため、`iptables`コマンドは別途インストールが必要です。

WireGuardの起動スクリプトが内部でiptablesを呼び出すため、これを知らずにいるとWireGuardが起動失敗します。エラーメッセージは`iptables: command not found`。シンプルなエラーですが、原因を知らないと詰まるポイントです。

### NIC名が`eth0`じゃない

古いLinuxでは`eth0`という名前だったネットワークインターフェース(NIC)が、Amazon Linux 2023では`ens5`という名前になっています。設定ファイルに`eth0`と書いてしまうとMASQUERADE(NATの一種)が空振りし、「VPNは繋がるのにその先に届かない」問題が発生します。

### `welcome.conf`の罠

ApacheのデフォルトエラーページをModSecurityが上書きしてくれると思いきや、`welcome.conf`というデフォルト設定ファイルが「403エラーが来たら`It works!`ページを200で返す」という挙動をしていました。

WAFが攻撃を検知して403を返しているのに、ブラウザには200と「It works!」が表示される──気づくまでにしばらくかかりました。

## まとめ

今回の構築を通じて感じたことをひとことで言うと、**「セキュリティは積み重ねで成り立っている」**ということです。

WAFだけでは防ぎきれない。iptablesだけでは見えない。VPNだけでは管理できない。それぞれが「自分の役割」を担い、組み合わさることで多層防御が成立します。

また、クラウド時代のセキュリティは「自前で全部実装する」のではなく、「マネージドサービスを使う部分」と「自分で実装する部分」を正しく見極めることも重要だと感じました。Suricataを諦めてGuardDutyで代替する、という判断はまさにその実例です。

次は、今回構築した環境に**ログ収集・可視化の仕組み(OpenSearch等)**を組み合わせて、「何が起きているかを一画面で見る」ことに挑戦してみたいと思います。
