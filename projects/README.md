## final-exercise

→ [`final-exercise/`](final-exercise/)

### 構成概要

Web / AP / DB / DNS / SMTP / NFS / NTP など、これまで学んだ技術を統合した総合構築をチームで実施した記録。

### 役割

- チーム全体の方向性を揃えるサポート役
- DNS / SMTP / NFS など特定サービスの主担当
- チームメンバーへの技術サポート・レビュー

### 学んだこと・ハマりどころ

複数サービスを同一環境に同居させるとき、構築そのものよりも **「順番」と「合意」** にコストがかかると体感した。

- **依存関係の順番設計**: DNS が先、SMTP がその次、NFS は OS ユーザが揃ってから、というように、サービス間の依存を踏まえた構築順序を設計する必要があった。順序を誤ると、後段のサービスで原因不明の失敗が連鎖する
- **設計・パラメータの意思統一**: ホスト名 / IP / ドメイン / ユーザ名 などをチーム内で揃える作業。一見地味だが、これが揃っていないと結合の段階で必ず破綻するため、設計書・パラメータ集約表を早めに整える重要性を学んだ
- **サービスをまたぐトラブル切り分け**: DNS ・ SMTP ・ NFS をまたぐ問題が発生したとき、どの層で起きているかを切り分けるのが難しかった。各サービスのログを横並びで追い、設定を一つずつ外しながら原因層を特定するアプローチで対応
- **チーム内の進捗同期とレベル差**: タスク分担と進捗の見える化、メンバー間の技術レベル差をどう埋めるかが課題だった。手を動かす前にホワイトボードで構成図を共有する、詰まったメンバーには手順より「今何を確認しているか」を一緒に整理する、といった進め方を試した

## nginx-tomcat-pg-nfs-redundancy

→ [`nginx-tomcat-pg-nfs-redundancy/`](nginx-tomcat-pg-nfs-redundancy/)

### 構成概要

Web-AP 層を冗長化した5台構成。Nginx ロードバランサ配下に AP サーバを複数台配置し、NFS でファイル共有、PostgreSQL で永続化する構成。
1台 → 3台 → 3台 + NFS → 5台冗長化、と段階的に積み上げた集大成。

### 学んだこと・ハマりどころ

「1台で動く構成」を素直に複数台に並べただけでは動かない、ということを各層で経験した。

- **Nginx ロードバランシング設定**: `upstream` に複数 AP を並べたときの振り分けアルゴリズム(`round-robin` / `ip_hash` など)の選定と、それに伴う **セッション保持をどの層で持たせるか**(Nginx / AP / DB)の設計判断。`proxy_pass` の指定、`Host` / `X-Forwarded-For` などのヘッダーを後段にどう受け渡すかも合わせて検討した
- **NFS によるファイル共有**:
  - AP サーバから NFS マウント先のファイルを操作した際に発生した **削除・書き込み権限エラー**。`root_squash` の挙動を理解していなかったことが原因で、`exports` のオプションと AP プロセスの実行ユーザ・UID/GID を揃える設計の重要性を学んだ
  - 複数 AP サーバから同時に NFS を読み書きする際の **レスポンス遅延**。マウントオプションや NFS の仕組み(同期/非同期、キャッシュ)を追いかけた

### 解決の進め方

- 設定を一つずつ切り分けて原因を特定(例: Nginx を外して AP 直接アクセスで切り分け、NFS をローカルディスクに置き換えて切り分け、など)
- アクセスログ / エラーログ / `journalctl` を横並びで追跡
- 公式ドキュメント(Nginx の `upstream` モジュール、`exports(5)` など)を都度参照し、自分の言葉で要約してから設定に落とす
