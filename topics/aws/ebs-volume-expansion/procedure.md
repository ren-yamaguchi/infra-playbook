## 2. 目的・概要

### 2-1. 目的

本手順書では、稼働中の EC2 インスタンスにアタッチされている EBS ボリュームの容量を拡張する。
拡張は以下の 2 段階で行う必要があるため、両方の作業を実施する。

1. **AWS 側での容量拡張**：EBS ボリューム自体のサイズを拡張する
2. **Linux 側での容量拡張**：拡張したサイズを OS（パーティション・ファイルシステム）に反映させる

> **注意：** AWS 側だけ拡張しても、OS からは元のサイズのまま見えるため、必ず Linux 側の作業も実施する必要がある。

### 2-2. 構成概要（アーキテクチャ）

```
[EC2 インスタンス]
   |
   ├── EBS ボリューム（拡張対象）
   |       ├── パーティション（必要に応じて拡張）
   |       └── ファイルシステム（必要に応じて拡張）
```

### 2-3. 完成イメージ（ゴール定義）

- [ ] AWS マネジメントコンソール上で EBS ボリュームのサイズが変更されている
- [ ] EC2 インスタンス上で `lsblk` を実行し、ボリュームサイズが拡張後の値になっている
- [ ] パーティションサイズがボリュームサイズと一致している
- [ ] ファイルシステムサイズがボリュームサイズと一致している（`df -hT` で確認）

## 4. 構築手順（詳細）

> **注意事項**
> - コマンド中の `<山カッコ>` は自分の環境の値に置き換えること
> - AWS マネジメントコンソール（ブラウザ上で AWS にログインして操作する画面）を以降「GUI 画面」と呼称する

------------------------------

### Step 1：AWS マネジメントコンソールでの容量拡張

**目的：** EBS ボリューム自体のサイズを AWS 側で拡張する。

#### 操作手順

1. GUI 画面で「EC2」を検索し、EC2 ダッシュボードに移動する。
2. 「インスタンス」一覧で、検索窓に自分のインスタンス名を入力し、対象のインスタンスを特定する。
3. 該当行の「インスタンス ID」をクリックし、インスタンス概要ページに移動する。
4. ページを下にスクロールし、タブ一覧から「ストレージ」をクリックする。
5. 「ブロックデバイス」の欄から、拡張対象のボリュームの「ボリューム ID」をクリックする。
   > **補足：** ボリュームが複数表示される場合は、右にスクロールして「アタッチ時刻」で対象を判別する。
6. ボリュームの画面に遷移したら、対象ボリューム左の □ にチェックを入れる。
7. 画面右上の「アクション ▼」をクリックし、「ボリュームの変更」を選択する。
8. 「サイズ (GiB)」の欄に拡張後のサイズを入力する。
9. 画面右下の「変更」ボタンをクリックする。
10. 確認ダイアログが表示されたら「変更」をクリックする。

> **確認：** ボリュームの「状態」が「in-use - optimizing」→「in-use - completed」に変化することを確認する（数分〜数十分かかる場合がある）。

------------------------------

### Step 2：Linux 側で現在の状態を確認

**目的：** Linux 側でボリューム・パーティション・ファイルシステムの現在のサイズを確認し、次にどの作業が必要かを判断する。

#### 操作手順

1. EC2 インスタンスに SSH 接続する。
2. ブロックデバイス（ボリューム・パーティション）の状態を確認する。
   ```
   lsblk
   ```
3. 出力例を確認し、拡張対象のボリュームを特定する。
   ```
   NAME          MAJ:MIN RM  SIZE RO TYPE MOUNTPOINT
   nvme0n1       259:0    0   20G  0 disk         ← ボリューム（拡張後のサイズ）
   └─nvme0n1p1   259:1    0   8G   0 part /       ← パーティション（拡張前のサイズ）
   ```
4. 上記の例のように **ボリュームサイズ > パーティションサイズ** となっている場合、パーティションの拡張が必要 → **Step 3 へ**。
5. ボリューム直下にパーティションが存在しない場合は、パーティション拡張は不要 → **Step 4 へ**。

> **確認：** `lsblk` の結果から、自分のボリューム名・パーティション名・現在のサイズを把握できていること。

------------------------------

### Step 3：パーティションの拡張

**目的：** Step 1 で拡張したボリュームに合わせてパーティションのサイズも拡張する。

#### 操作手順

1. `growpart` コマンドでパーティションを拡張する。
   ```
   sudo growpart <デバイス名> <パーティション番号>
   ```
   **実行例**（`/dev/nvme0n1` の 1 番目のパーティションを拡張する場合）：
   ```
   sudo growpart /dev/nvme0n1 1
   ```
   > **注意：** デバイス名とパーティション番号の間は **半角スペース** で区切る。`/dev/nvme0n1p1` のように一体で書かない。
2. 再度 `lsblk` で確認する。
   ```
   lsblk
   ```

> **確認：** ボリュームサイズとパーティションサイズが一致していることを確認する。

------------------------------

### Step 4：ファイルシステムの拡張要否を確認

**目的：** ファイルシステムのサイズがボリュームサイズより小さい場合に拡張する必要があるため、現状を確認する。

#### 操作手順

1. ファイルシステムのサイズとタイプを確認する。
   ```
   df -hT
   ```
   > **補足：** `-T` オプションでファイルシステムのタイプ（xfs / ext4 など）が表示される。
2. 出力例：
   ```
   Filesystem     Type   Size  Used Avail Use% Mounted on
   /dev/nvme0n1p1 xfs    8.0G  2.0G  6.0G  25% /
   ```
3. **ボリュームサイズ > ファイルシステムサイズ** となっている場合 → **Step 5 へ**。
4. ボリュームサイズとファイルシステムサイズが一致している場合 → 作業完了。

> **確認：** ファイルシステムのタイプ（xfs / ext4）と、現在のサイズ・マウントポイントを把握できていること。

------------------------------

### Step 5：ファイルシステムの拡張

**目的：** ファイルシステムを拡張し、OS からボリュームの全容量を使えるようにする。ファイルシステムのタイプによってコマンドが異なるため、Step 4 で確認した「Type」に応じて実行する。

#### 操作手順（xfs の場合）

1. マウントポイントを指定して `xfs_growfs` を実行する。
   ```
   sudo xfs_growfs -d <マウントポイント>
   ```
   **実行例**（マウントポイントが `/` の場合）：
   ```
   sudo xfs_growfs -d /
   ```

#### 操作手順（ext4 の場合）

1. デバイス名（パーティション名）を指定して `resize2fs` を実行する。
   ```
   sudo resize2fs <パーティションのデバイスパス>
   ```
   **実行例**（パーティションが `/dev/nvme0n1p1` の場合）：
   ```
   sudo resize2fs /dev/nvme0n1p1
   ```

#### 拡張後の確認

1. 再度ファイルシステムのサイズを確認する。
   ```
   df -hT
   ```

> **確認：** ファイルシステムサイズがボリュームサイズと一致していることを確認する。

## 付録（任意）

### A. コマンド解説

| コマンド | 説明 |
|---------|------|
| `lsblk` | ブロックデバイス（ボリューム・パーティション）の一覧とサイズを表示する。 |
| `df -hT` | マウントされているファイルシステムの使用状況とタイプを人間が読みやすい形式で表示する。 |
| `sudo growpart <デバイス> <番号>` | 指定したパーティションを、利用可能な最大サイズまで拡張する。 |
| `sudo xfs_growfs -d <マウントポイント>` | xfs ファイルシステムをデバイスの最大サイズまで拡張する。 |
| `sudo resize2fs <デバイスパス>` | ext2/ext3/ext4 ファイルシステムをデバイスのサイズに合わせて拡張する。 |

### C. 用語解説

| 用語 | 説明 |
|------|------|
| EBS | Elastic Block Store。EC2 にアタッチして使えるブロックストレージサービス。 |
| ボリューム | EBS のストレージ単位。EC2 から見ると 1 つのディスクとして認識される。 |
| パーティション | ボリュームを論理的に分割した領域。1 つのボリュームに複数のパーティションを作れる。 |
| ファイルシステム | ファイルを管理する仕組み。xfs、ext4 などの種類がある。 |
| マウントポイント | ファイルシステムを OS のディレクトリツリーに紐づける場所（例：`/`、`/data`）。 |

------------------------------

### D. 補足解説

- **なぜ 3 段階（ボリューム → パーティション → ファイルシステム）の拡張が必要なのか**
  EBS ボリュームは「物理的な箱（ボリューム）」→「箱の中の仕切り（パーティション）」→「中身の整理ルール（ファイルシステム）」という階層構造になっている。AWS 側で箱を大きくしても、内側の仕切りや整理ルールは自動では広がらないため、それぞれ個別に拡張する必要がある。

- **パーティションがない構成について**
  ボリューム直下にファイルシステムを作っている構成（パーティションを切らずにボリューム全体を 1 つのファイルシステムとして使用）の場合、`lsblk` の出力でボリューム配下にパーティションが表示されない。この場合は Step 3（パーティション拡張）はスキップし、Step 4（ファイルシステム確認）に進む。

- **作業前のスナップショット取得を推奨**
  パーティションやファイルシステム操作は万が一の場合データ消失のリスクがある。本番環境で作業する場合は、必ず事前に EBS スナップショットを取得しておくこと。
