## 2. 目的・概要

### 2-1. 目的

> 本手順書では、AWS上に2台のEC2インスタンスとALB(Application Load Balancer)を構築し、Apacheによるアクセスの負荷分散環境を構築する。
> 構築後はALBによる(1)均等な負荷分散、(2)パスルーティング、(3)ヘルスチェック専用ログ分離 が動作する状態を目指す。

### 2-2. 構成概要(アーキテクチャ)

```
                    [ローカルPC ブラウザ]
                            |
                            | HTTP (80)
                            v
                          [ALB]
                       /         \
                      /  ルーティング
                     v             v
            [EC2 サーバA]      [EC2 サーバB]
              Apache             Apache
              /saba/             /miso/
              /healthcheck/      /healthcheck/
```

- **ALB**: ローカルPCからのHTTPアクセスを2台のEC2に振り分ける
- **EC2 サーバA / サーバB**: Apacheで `/saba/` または `/miso/` のコンテンツを提供。`/healthcheck/` でヘルスチェックを受ける(別アベイラビリティーゾーンに配置)

### 2-3. 完成イメージ(ゴール定義)

- [ ] ALBのDNS名でアクセスしたとき、2台のEC2にアクセスが均等に割り振られている
- [ ] パス名(`/saba/` / `/miso/`)によってアクセスの振り分けができている
- [ ] ヘルスチェック専用のログファイルにヘルスチェックログが記録され、通常のアクセスログには記録されていない

### Step 1: セキュリティグループの設定

**目的:** EC2用とALB用のセキュリティグループを作成する。

#### EC2用セキュリティグループ

| タイプ | プロトコル | ポート範囲 | ソース | 説明 |
|--------|-----------|-----------|--------|------|
| SSH | TCP | 22 | マイIP | ローカルPCからSSHログイン |
| HTTP | TCP | 80 | ALBのセキュリティグループ | ALBからのHTTP接続を許可するため |

> **ポイント:** ALBからのHTTP接続を許可しないとタイムアウトになる。
> 通常のEC2構成では外部からのリクエストをEC2が直接受け取るが、ALB構成では外部からのリクエストをALBが受け取り、ALBからEC2へHTTPリクエストを送るため、EC2側のSGでALBからのHTTP接続を許可しておく必要がある。

#### ALB用セキュリティグループ

| タイプ | プロトコル | ポート範囲 | ソース | 説明 |
|--------|-----------|-----------|--------|------|
| HTTP | TCP | 80 | マイIP | ローカルPCのブラウザからALBへ接続 |

### Step 2: EC2インスタンスの起動

**目的:** 別アベイラビリティーゾーン(AZ)に配置した2台のEC2インスタンスを作成する。

#### 操作手順

通常の起動と異なる設定のみ記載:

- ネットワーク設定の「編集」を開き、**サブネットをペアのメンバーと別のAZ** に変更する
  - サブネットを選択すると自動的にAZも選択される
- 2台のEC2は **異なるAZのサブネット** に配置する

### Step 3: HTTPサーバ(Apache)の構築

**目的:** 各EC2にApacheをインストールし、HTTPサーバとして起動する。**サーバA・サーバBの両方で実施**する。

#### 操作手順

```bash
# ローカルPCからSSHログイン
ssh -i <秘密鍵のファイルパス> ec2-user@<EC2のパブリックIP>

# rootユーザーにスイッチ
sudo su -

# システムの時間を日本時間に設定
timedatectl set-timezone Asia/Tokyo

# 最新パッケージへの更新
dnf update -y

# Apacheをインストール
dnf install -y httpd

# Apacheの起動と自動起動設定
systemctl enable --now httpd

# Apacheの起動確認
systemctl status httpd | less

# Apacheの自動起動設定確認
systemctl is-enabled httpd
```

### Step 6: アクセスログの確認(均等負荷分散)

**目的:** ALBのDNS名でアクセスし、2台のEC2にアクセスが均等に割り振られていることを確認する。

#### 操作手順

ブラウザで `http://<ALBのDNS名>` を検索し、各EC2のApacheアクセスログでアクセスが均等に振られていることを確認する。

```bash
# 各EC2でApacheのアクセスログをリアルタイムで確認
tail -f /var/log/httpd/access_log
```

アクセスログの例:
```
172.31.12.154 - - [09/May/2026:14:17:28 +0900] "GET / HTTP/1.1" 403 191 "-" "Mozilla/5.0 ..."
172.31.42.233 - - [09/May/2026:14:28:01 +0900] "GET / HTTP/1.1" 403 191 "-" "Mozilla/5.0 ..."
```

> **補足:** ステータスコード 403/404 が出る理由については、付録D「アクセスログの一歩先の知識」を参照。

### Step 7: パスルーティング設定

**目的:** パス名(`/saba/` / `/miso/`)によって、サーバA/Bに振り分ける設定を行う。

#### 7-1. EC2側(サーバA・サーバB)の設定

`/var/www/html/` とその配下の全てのファイル・サブディレクトリをApacheが操作できるようにする。

**サーバAの設定:**

```bash
# /var/www/html/と配下の所有グループをapacheに変更
chown -R :apache /var/www/html/

# /var/www/html/配下に作成されるファイル・サブディレクトリの所有グループを親と同じにし、
# 所有グループに全権限を付与するためSGID「2」を設定
chmod 2775 /var/www/html/

# 所有グループと権限の確認
ll /var/www/

# 既に/var/www/html/配下にサブディレクトリが存在する場合
find /var/www/html -type d -exec sudo chmod 2775 {} +

# 既に/var/www/html/配下にファイルが存在する場合
find /var/www/html -type f -exec sudo chmod 0664 {} +

# /var/www/html/配下にsabaディレクトリを作成
mkdir /var/www/html/saba

# /var/www/html/saba/配下にsaba.htmlを作成
vi /var/www/html/saba/saba.html
```

saba.htmlの内容:

```html
<!DOCTYPE html>
<html>
<head>
    <title>Saba</title>
</head>
<body>
    saba
</body>
</html>
```

`http://<ALBのDNS名>/saba/` にアクセスした時に `/saba/saba.html` の内容を表示するように、Apache設定ファイルに追記する。

```bash
# Apacheの設定ファイルのバックアップを取得(原本保存)
cp /etc/httpd/conf/httpd.conf{,.org}

# Apacheの設定ファイルに追記
vi /etc/httpd/conf/httpd.conf
```

設定ファイルの編集内容:

```
# ---ファイル末尾に追記---
<Directory "/var/www/html/saba">
    DirectoryIndex saba.html
</Directory>
# ------------------------
```

```bash
# 設定ファイルを反映させるため再起動
systemctl restart httpd
```

**サーバBの設定:** 上記の `/saba/` を `/miso/` に、`saba.html` を `miso.html` に、HTMLの中身も `saba` を `miso` に変更して同じ手順を実施する。

#### 7-2. ALBのリスナールール設定(GUI操作)

**サーバA向けの設定:**

`ALBの画面 → リスナーとルール → 設定したルールを選択 → ルールを管理 → ルールを追加する`

条件 → 条件の追加 → パスを選択:

| 一致パターンタイプ | パス条件値 |
|-------------------|------------|
| 値マッチング | `/saba/*` |

`アクション → アクションのルーティング → ターゲットグループへ転送 → サーバAの属するターゲットグループを選択`

「次へ」を選択。

> **補足:** リスナールールの優先度は数字が小さいほど高い。今回の場合は2台のサーバに異なるパスで振り分けるので、どちらの優先度が高くても問題ない。

`リスナールール → 優先度を任意の数字で記入`

「次へ」を選択し、確認画面で「ルールを追加する」を選択。

**サーバB向けの設定:** パス条件値を `/miso/*` に設定し、優先度を別の数字にして同じ手順を実施。

#### 7-3. アクセスの振り分け確認

ブラウザで `http://<ALBのDNS名>/saba/` または `http://<ALBのDNS名>/miso/` を検索し、各サーバのアクセスログで正しく振り分けが行われていることを確認する。

```bash
# Apacheのアクセスログをリアルタイムで確認
tail -f /var/log/httpd/access_log
```

成功例(`/saba/` 検索時、サーバA):
```
172.31.12.154 - - [09/May/2026:15:04:31 +0900] "GET /saba/ HTTP/1.1" 200 95 "-" "Mozilla/5.0 ..."
```

失敗例(`/saba/` 検索時、サーバBに振られた場合):
```
172.31.42.233 - - [09/May/2026:14:57:53 +0900] "GET /saba/ HTTP/1.1" 404 236 "-" "Mozilla/5.0 ..."
```

ステータスコードと記録されているサーバから振り分けの成否を判定できる。詳細解説は付録D「アクセスログの一歩先の知識」参照。

### Step 8: ヘルスチェック設定とログ分離

**目的:** ALBからのヘルスチェックを設定し、ヘルスチェックログを通常のアクセスログと別ファイルに分離する。

> Step 7の権限変更は既に完了しているため、ここでは省略。

具体的な設定は以下の3つ:
1. ヘルスチェック専用のページを作成
2. (Step 7で完了済みのためスキップ)
3. ヘルスチェックログを通常のアクセスログと別ファイルに保存

**サーバA・サーバB両方で実施:**

```bash
# ヘルスチェック先ファイルを置くディレクトリの作成
mkdir /var/www/html/healthcheck

# ヘルスチェック先のファイルの作成
touch /var/www/html/healthcheck/healthcheck.html
```

> **解説:** ターゲットグループ作成時のヘルスチェックパスと同じパスのファイル・ディレクトリを作成する必要がある。
> 例) ヘルスチェックパスが `/check.html` の場合、ディレクトリ作成は不要で `/var/www/html/check.html` を作るだけで済む。

```bash
# ヘルスチェックのログを保存するファイルを作成
touch /var/log/httpd/healthcheck_log

# ヘルスチェックのログを別の場所に保存するためにApacheの設定ファイルを編集
vi /etc/httpd/conf/httpd.conf
```

設定ファイルの編集内容:

```
# ---<IfModule log_config_module>セクション内の
# <IfModule logio_module>セクションの下に以下を追記---
SetEnvIf Request_URI "^/healthcheck/" is_healthcheck_dir
CustomLog "logs/healthcheck_log" combined env=is_healthcheck_dir
CustomLog "logs/access_log" combined env=!is_healthcheck_dir

# 既存のアクセスログの保存場所設定をコメントアウト
#CustomLog "logs/access_log" combined
# -------------------------------------------------
```

```bash
# 設定を反映するため再起動
systemctl restart httpd
```

## 6. トラブルシューティング

### よくあるエラーと対処法

---

#### エラー①: ブラウザで `http://<ALBのDNS名>` にアクセスしてもタイムアウトする

**原因:** EC2側のSGで、ALBのSGからのHTTPアクセスが許可されていない

**対処法:** EC2のSGインバウンドルールに以下を追加:

| タイプ | ポート | ソース |
|--------|-------|--------|
| HTTP | 80 | ALB用SG |

---

#### エラー②: ALBのターゲットグループのヘルスチェックが「unhealthy」になる

**原因:** ヘルスチェックパスのファイルが存在しない、または Apache が起動していない、SG設定ミス

**対処法:**
```bash
# ヘルスチェック用ファイルが存在するか確認
ls -l /var/www/html/healthcheck/healthcheck.html

# Apacheの状態確認
systemctl status httpd

# Apacheのアクセスログ・エラーログを確認
tail -f /var/log/httpd/access_log
tail -f /var/log/httpd/error_log
```

---

#### エラー③: パスルーティングが想定通り動作しない

**原因:** リスナールールの優先度設定ミス、`Directory` ディレクティブの記述ミス

**対処法:**
- ALBのリスナールール画面で、`/saba/*` → サーバAのターゲットグループ、`/miso/*` → サーバBのターゲットグループの順序になっているか確認
- 各サーバの `httpd.conf` の `<Directory>` 記述を確認
- Apache設定の構文チェック: `httpd -t`

---

### ログの確認場所

| ログの種類 | 場所(パス) | 確認コマンド |
|-----------|------------|------------|
| Apacheアクセスログ | `/var/log/httpd/access_log` | `sudo tail -f /var/log/httpd/access_log` |
| Apacheエラーログ | `/var/log/httpd/error_log` | `sudo tail -f /var/log/httpd/error_log` |
| ヘルスチェックログ | `/var/log/httpd/healthcheck_log` | `sudo tail -f /var/log/httpd/healthcheck_log` |
| ALBアクセスログ(任意) | S3バケット(別途設定) | ALBアクセスログ有効化が必要 |

## 付録(任意)

### A. コマンド解説

#### A-1. `chown -R <所有ユーザー>:<所有グループ> <ディレクトリパス>`

指定したディレクトリとその中身全てに対して、所有ユーザーと所有グループを一括変更する。

- `chown`: Change Owner の略。所有権を変更する
- `-R` (Recursive): 指定ディレクトリ自体と、その下の全サブディレクトリ・ファイルにも同じ変更を適用
- `<所有ユーザー>:<所有グループ>`: `:`(コロン)で区切って指定
- `<ディレクトリパス>`: 変更対象

#### A-2. `find <ディレクトリパス> -type <タイプ> -exec chmod <権限> {} +`

特定条件(ファイル/ディレクトリ)に一致する項目だけを検索し、一括で権限を変更する。

- `find <ディレクトリパス>`: 指定ディレクトリ内を再帰的に検索
- `-type <タイプ>`: 検索対象の種類
  - `f`: ファイルのみ
  - `d`: ディレクトリのみ
  - `l`: シンボリックリンクのみ
- `-exec chmod <権限> {} +`: 見つかった対象に chmod を実行
  - `{}`: find で見つかったパスがここに入る
  - `+`: 見つかった複数のパスをまとめて1回のコマンドとして実行(`\;` だと1つずつ実行)

> **注意:** root以外の場合は `chmod` の前に `sudo` が必要。

#### A-3. `find /var/www/html -type f -exec sudo chmod 0664 {} +` で権限を `0664` にする理由

これがセキュリティ上のベストプラクティスだから:

- **実行権限(x)は不要**: HTMLや画像、PHPスクリプトなどのWebファイルは「Apacheが中身を読み取る」だけのもの。OS上で直接実行するものではない
- **リスク回避**: 万が一悪意のあるファイルがアップロードされた際、実行権限が付いているとシェルスクリプトとして実行されサーバ乗っ取りのリスクが高まる

### B. 設定ファイル解説(Apache)

#### B-1. ディレクトリにアクセスされた時の優先ファイルの表示

```
<Directory "<ディレクトリパス(末尾にスラッシュは不要)>">
    DirectoryIndex <ディレクトリ内のファイル>
</Directory>
```

特定のディレクトリにアクセスがあった際に優先的に表示するファイル(インデックスファイル)を指定する設定。

- `<Directory "<ディレクトリパス>">`: 指定したディレクトリに対してのみ設定を適用する範囲(コンテキスト)を定義。パスの末尾にスラッシュが不要なのはApacheの内部処理で正規化されるため
- `DirectoryIndex <ファイルパス>`: そのディレクトリがリクエストされた時にサーバが探しに行くファイル名
  - 例) `DirectoryIndex index.php main.html` と書くと、まず `index.php` を探し、なければ `main.html` を表示する

**注意点:**
- `mod_dir`: この機能を利用するには Apacheで `mod_dir` モジュールが有効である必要がある
- 末尾のスラッシュ(DirectorySlash): ブラウザからスラッシュなしでディレクトリにアクセスした際、Apacheは通常末尾にスラッシュを付けたURLへリダイレクトする
- 記述場所: 通常は `httpd.conf` や `apache2.conf`、または各サイトの設定ファイルに記述

#### B-2. 内部変数によるログの保存先分岐

```
SetEnvIf Request_URI "<ディレクトリパス>" <内部変数>
CustomLog "<ファイルパス>" combined env=<内部変数>
CustomLog "<ファイルパス>" combined env=!<内部変数>
```

特定の条件に基づいてログの出力先を振り分ける(または除外する)設定。

- `SetEnvIf Request_URI "<ディレクトリパス>" <内部変数>`: アクセスURLが指定パスに合致する場合、`<内部変数>` をオンにする
- `"<ディレクトリパス>"` は正規表現
- `CustomLog "<ファイルパス>" combined env=<内部変数>`: `<内部変数>` に合致するアクセスを指定ファイルに記録
- `CustomLog "<ファイルパス>" combined env=!<内部変数>`: `<内部変数>` に合致しないアクセスを指定ファイルに記録

**補足:** `combined` はログの出力フォーマット(記録する情報の種類)を指定する標準的な形式。

### C. 用語解説

#### C-1. SUID (Set User ID)

実行ファイルに設定される権限で、プログラムを実行したユーザーではなく「ファイルの所有者」の権限で実行させる仕組み。

- 役割: 一般ユーザーが、一時的に管理者(root)などの強力な権限を借りて特定の作業を行うために使う
- 代表例: `/usr/bin/passwd` コマンド(パスワード変更は本来root権限が必要な `/etc/shadow` を書き換えるが、SUIDのおかげで一般ユーザーも自分のパスワードを変更できる)
- 見分け方: `ls -l` で所有者の実行権限が `s` になる(例: `-rwsr-xr-x`)
- 設定: `chmod u+s <ファイルパス>` または `chmod 4755 <ファイルパス>`

#### C-2. SGID (Set Group ID)

ファイルやディレクトリに設定される特殊なアクセス権限。

- **実行ファイルに設定した場合:** 「ファイルの所有グループ」の権限で動作する
- **ディレクトリに設定した場合:** そのディレクトリ内で新しく作成されたファイル・サブディレクトリが、作成者の主グループではなく親ディレクトリの所有グループを自動的に継承する

#### C-3. スティッキービット (Sticky Bit)

主に共有ディレクトリに設定される権限で、「自分(所有者)が作ったファイル以外は削除・名前変更できない」ように制限する仕組み。

- 役割: 誰でもファイルを書き込める場所(共有フォルダ)で、他人のファイルを勝手に消されないように保護
- 代表例: `/tmp` ディレクトリ
- 見分け方: `ls -l` でその他のユーザーの実行権限が `t` になる(例: `drwxrwxrwt`)
- 設定: `chmod o+t <ディレクトリパス>` または `chmod 1777 <ディレクトリパス>`

#### C-4. 特殊権限まとめ

| 権限 | 数値 | 対象 | 主な効果 |
|------|------|-----|----------|
| SUID | 4000 | ファイル | 所有者の権限で実行する |
| SGID | 2000 | ディレクトリ | 所有グループを継承させる |
| スティッキー | 1000 | ディレクトリ | 他人のファイルを削除不能にする |

### D. アクセスログの一歩先の知識

#### D-1. 初回アクセス時のステータスコード 403 / 404

ブラウザで `http://<ALBのDNS名>` を検索すると、初回は以下のように同時刻のログが2種類記録される。

```
172.31.12.154 - - [09/May/2026:14:17:28 +0900] "GET / HTTP/1.1" 403 191 "-" "Mozilla/5.0 ..."
172.31.12.154 - - [09/May/2026:14:17:28 +0900] "GET /favicon.ico HTTP/1.1" 404 236 "..." "Mozilla/5.0 ..."
```

- **403のログ:** `index.html` が存在しない場合、Apacheはステータスコード403を返す。`index.html` を作成しない限り記録され続ける
- **404のログ:** モダンブラウザ(Chrome、Safari等)はサイトのタブに表示するアイコン用に、サーバのルートディレクトリにある `favicon.ico` を自動でリクエストするが、これも作成していないため404を返す

2回目以降は、ブラウザが「このサイトにアイコンはない」という情報をキャッシュ(無駄な通信を減らすため)するため、404ログは記録されず、403のみが記録され続ける。

#### D-2. パスルーティング時のステータスコード解説

**成功例(`/saba/` 検索 → サーバAに振られた場合):**

`"GET /saba/ HTTP/1.1" 200` がサーバAのアクセスログで確認できる。
ALBがALB → サーバAへの振り分けに成功し、`/saba/` へのアクセスも成功した状態。

```
"GET /saba/ HTTP/1.1" 304
```
2回目以降は304(Not Modified)が返る。コンテンツが更新されていないのでブラウザのキャッシュからコンテンツが返された状態。

**失敗例(`/saba/` 検索なのにサーバBに振られた場合):**

サーバBのアクセスログに `"GET /saba/ HTTP/1.1" 404` が記録される。
これはALBがサーバBに振り分けてしまい、サーバBには `/saba/` がないため404を返した状態。

#### D-3. ヘルスチェックログの分離成否

| ステータスコード | 記録先 | 状態 |
|----------------|--------|------|
| 200 | healthcheck_log | **成功**: ログ分離も、ヘルスチェックも両方成功 |
| 404 | healthcheck_log | ログ分離は成功、ヘルスチェック用ファイル未作成 |
| 200 | access_log | ヘルスチェックは成功、ログ分離設定に失敗(`SetEnvIf`/`CustomLog` 設定ミス) |
| 404 | access_log | ログ分離もファイル配置も両方失敗 |

ヘルスチェックログ(`ELB-HealthChecker/2.0` というUserAgentが特徴)が `healthcheck_log` に200で記録されていれば、ヘルスチェック設定とログ分離設定の両方が成功している。
