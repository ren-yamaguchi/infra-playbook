## 基本課題 A：インストールと基本設定

**A-1. EC2 への Squid インストール**
- AL2023 の EC2 に Squid をインストールし、`systemd` でサービス登録・自動起動を設定する
- `squid -v` でビルドオプションを確認し、`squid -k check` で設定ファイルの文法チェックを行う
- デフォルトポート（3128）でプロキシが動作することを確認し、クライアント EC2 から `curl -x http://10.0.1.100:3128 http://example.com` でアクセスできることを確認する

**A-2. squid.conf の基本設定**
- `http_port`・`cache_dir`・`cache_mem`・`access_log`・`cache_log` を設定する
- `http_access deny all` がデフォルトで設定されているため、許可する ACL を明示的に追加しないとアクセスできないことを確認する

## 基本課題 B：アクセス制御リスト（ACL）

**B-1. IP ベースのアクセス制御**
- 特定の IP アドレスからのみプロキシ経由のアクセスを許可する ACL を設定する

```squid
acl localnet src 10.0.0.0/16
http_access allow localnet
http_access deny all
```

**B-2. 時間帯によるアクセス制御**
- `acl workhours time MTWHF 09:00-18:00` で業務時間のみアクセスを許可し、時間外はブロックする設定を行う

**B-3. ドメインベースのアクセス制御**
- `acl blocked_sites dstdomain .example-blocked.com` で特定ドメインへのアクセスをブロックする設定を行い、ブロック時にカスタムエラーページを表示する

## 基本課題 C：キャッシュの設定

**C-1. ディスクキャッシュの設定**
- `cache_dir ufs /var/spool/squid 1000 16 256` で 1GB のディスクキャッシュを設定する
- `squid -z` でキャッシュディレクトリを初期化し、キャッシュが動作していることを `squid -k stat` で確認する

**C-2. キャッシュヒットの確認**
- 同じ URL に 2 回アクセスし、2 回目のレスポンスに `X-Cache: HIT from` ヘッダーが含まれることを確認する
- `squidclient mgr:info` でキャッシュヒット率を確認する

## 発展課題 D：URL フィルタリング

**D-1. ブラックリストファイルによるフィルタリング**
- ブロックするドメインのリストをファイルで管理し、`acl blocked_sites dstdomain "/etc/squid/blocked_domains.txt"` で読み込む設定を行う
- スクリプトでブラックリストを定期更新し、`squid -k reconfigure` でリロードする仕組みを構築する

**D-2. 正規表現によるURL フィルタリング**
- `acl bad_url urlpath_regex -i \.exe$ \.bat$` で実行ファイルのダウンロードをブロックする設定を行う

## 発展課題 E：認証の設定

**E-1. Basic 認証の設定**
- `htpasswd` で認証ファイルを作成し、Squid の `auth_param basic` で Basic 認証を設定する
- 認証済みユーザーのみプロキシを使用できる設定を行い、未認証アクセスが 407 で拒否されることを確認する

---

## 発展課題 F：SSL バンプ（HTTPS インスペクション）

**F-1. SSL バンプの設定**
- 自己署名 CA 証明書を生成し、HTTPS 通信を Squid でインターセプトして内容を検査する SSL バンプを設定する
- クライアントに自己署名 CA を信頼させ、HTTPS サイトへのアクセスが Squid のログに記録されることを確認する
- プライバシーとセキュリティ上の考慮点（バンクサイト等の個人情報が検査される問題）をまとめる

---

## 発展課題 G：監視・ログ分析

**G-1. アクセスログの分析**
- `access.log` を `awk` で解析し、アクセス頻度の高いサイト・帯域を多く使用するクライアントを集計するスクリプトを作成する
- `sarg`（Squid Analysis Report Generator）でアクセスログを HTML レポートに変換する

**G-2. CloudWatch との統合**
- Squid のアクセスログを CloudWatch Logs に転送し、ブロックされたアクセス数を時系列グラフで可視化する
- ブロックされたアクセスが急増した場合にアラームを発火させる設定を行う

---

## 発展課題 H：NAT インスタンスとの比較

**H-1. 比較と使い分けの整理**
- プライベートサブネットの EC2 からインターネットへのアクセスを制御する方法として、NAT インスタンス・NAT ゲートウェイ・Squid フォワードプロキシを比較する

| 観点 | NAT インスタンス | NAT ゲートウェイ | Squid |
|------|--------------|--------------|-------|
| アクセス制御 | IP / ポートのみ | IP / ポートのみ | URL・ドメイン・コンテンツ |
| ログ取得 | VPC Flow Logs | VPC Flow Logs | 詳細なアクセスログ |
| 管理コスト | 高 | 低（マネージド） | 高 |
| 料金 | EC2 料金 | 従量課金 | EC2 料金 |
